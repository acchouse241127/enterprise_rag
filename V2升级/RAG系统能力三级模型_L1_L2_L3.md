# RAG 系统能力三级模型（L1 / L2 / L3）

> **文档版本**：v1.0
> **编制日期**：2026-02-23
> **编制角色**：RAG 系统架构师 / RAG 产品经理
> **用途**：作为 Enterprise RAG 系统演进的理论框架，后续 PRD V2、TRD V2、开源项目评估均引用本文档的分级定义

---

## 一、分级核心逻辑

分级的本质不是"做什么业务"，而是**系统在多大程度上能自主地、可靠地完成从问题到答案的全过程**。

| 维度 | L1 基础级 | L2 专业级 | L3 自治级 |
|------|----------|----------|----------|
| **核心特征** | 检索+生成，线性流水线 | 多路智能检索+质量保障+场景适配 | 自主决策+自我纠错+持续进化 |
| **类比** | 一个会查资料的实习生 | 一个经验丰富的专业顾问 | 一个能自我学习的专家团队 |
| **能回答的问题** | "X是什么？" | "X和Y的关系是什么？为什么？" | "基于A、B、C三份文档的矛盾点，给出最优方案" |
| **错误处理** | 检索不到就说不知道 | 检索不到会换策略重试 | 能识别自己的错误并自动修正 |
| **适用场景** | FAQ、内部知识查询 | 客服、培训、合规咨询 | 金融分析、法律推理、研究辅助 |

## 二、L1 基础级：Naive RAG

### 2.1 能力定义

> 给定一个问题，从知识库中检索相关片段，拼接成 Prompt 交给 LLM 生成答案。流程固定，不会自我调整。

### 2.2 架构全景

```
用户问题
  ↓
[查询预处理] → 停用词/同义词扩展（规则）
  ↓
[单路/双路检索] → 向量检索 + 可选关键词检索
  ↓
[后处理] → 去重 + Reranker 重排序
  ↓
[Prompt 组装] → System Prompt + 检索上下文 + 用户问题
  ↓
[LLM 生成] → 流式输出
  ↓
[引用标注] → 后置插入 [ID:x]
  ↓
返回答案
```

### 2.3 完整技术栈

#### 后端

| 组件 | 技术选型 | 理由 |
|------|---------|------|
| **语言** | Python 3.11+ | RAG/ML 生态最成熟，库最全 |
| **Web 框架** | FastAPI | 异步原生、自动 OpenAPI 文档、性能优秀 |
| **任务队列** | Celery + Redis | 文档解析异步化，避免阻塞 API |
| **关系数据库** | PostgreSQL 16 | 成熟稳定，JSONB 支持灵活扩展 |
| **向量数据库** | ChromaDB 或 Qdrant | ChromaDB 轻量易上手；Qdrant 性能更好 |
| **ORM** | SQLAlchemy 2.0 | Python 标准 ORM |
| **缓存** | Redis | 会话管理、任务队列 |

#### 前端

| 组件 | 技术选型 | 理由 |
|------|---------|------|
| **框架** | React 18 + TypeScript | 生态最大，组件库丰富 |
| **构建** | Vite 5 | 开发体验好，HMR 快 |
| **UI 库** | Ant Design 或 Shadcn/ui | 企业级组件开箱即用 |
| **状态管理** | Zustand 或 React Query | 轻量，适合 API 驱动的应用 |
| **Markdown 渲染** | react-markdown + remark-gfm | 渲染 LLM 输出 |
| **SSE 客户端** | EventSource 或 fetch + ReadableStream | 流式输出 |

#### 模型矩阵

| 环节 | 模型 | 规格 | 部署方式 |
|------|------|------|---------|
| **Embedding** | BGE-M3（BAAI） | 1024维，中英双语 | 本地 GPU（推荐）或 CPU |
| **Reranker** | BGE-Reranker-v2-M3 | Cross-Encoder | 本地 GPU（推荐）或 CPU |
| **生成 LLM** | DeepSeek-V3 / GPT-4o-mini / Qwen2.5 | 通过 API 调用 | 云端 API |
| **查询扩展** | 同上 LLM 或规则 | - | 规则优先，LLM 可选 |

#### 算法与技术

| 环节 | 算法/技术 | 说明 |
|------|----------|------|
| **文本分块** | 滑动窗口 + 句子边界 | 按标点/段落切分，保持语义完整 |
| **向量检索** | HNSW（近似最近邻） | ChromaDB/Qdrant 内置 |
| **相似度度量** | Cosine Similarity | Embedding 空间的标准度量 |
| **关键词检索** | 词频重叠评分 | 简单有效的补充召回 |
| **去重** | SimHash + 汉明距离 | 近似重复检测 |
| **重排序** | Cross-Encoder 打分 | 精排，提升 top-K 质量 |
| **引用匹配** | 句子-Chunk 余弦相似度 | 后置匹配，插入引用标记 |

### 2.4 L1 的局限性

- 流程固定，不能根据问题类型自适应
- 只能处理单跳问题（"X是什么"），不能处理需要推理的问题
- 没有答案质量验证，LLM 幻觉无法检测
- 检索失败只能说"不知道"，不会尝试其他策略
- 所有知识库、所有用户、所有问题走同一条路

## 三、L2 专业级：Advanced RAG

### 3.1 能力定义

> 能根据问题的类型和复杂度，自动选择最优的检索策略和生成策略。具备质量保障能力，能检测并抑制幻觉。支持多场景适配。

### 3.2 与 L1 的本质区别

L1 是"给什么答什么"，L2 是"想清楚再答"。具体体现在：

1. **检索前**：理解问题意图，决定怎么检索
2. **检索中**：多策略并行，智能融合
3. **生成后**：验证答案质量，不合格就重来

### 3.3 架构全景

```
用户问题
  ↓
[意图分析层]
  ├─ 问题分类（FAQ/操作指引/分析比较/事实查询）
  ├─ 复杂度评估（单跳/多跳/需推理）
  └─ 场景路由 → 选择 Pipeline 配置
  ↓
[智能查询层]
  ├─ 查询改写（口语→检索语言）
  ├─ 查询分解（复杂问题拆成子问题）
  └─ HyDE（假设性文档生成，用于增强检索）
  ↓
[多路检索层]
  ├─ 密集检索（向量）
  ├─ 稀疏检索（BM25）
  ├─ 结构化检索（元数据过滤）
  └─ 知识图谱检索（实体关系，可选）
  ↓
[融合与精排层]
  ├─ RRF（Reciprocal Rank Fusion）多路融合
  ├─ Cross-Encoder Reranker
  ├─ 上下文压缩（去除无关段落）
  └─ 动态 top-K（根据分数分布自适应截断）
  ↓
[自适应生成层]
  ├─ 场景化 Prompt 模板
  ├─ 多轮对话上下文管理
  ├─ 结构化输出（JSON Schema 约束）
  └─ 流式 + 非流式双模式
  ↓
[质量保障层] ← L2 的核心新增
  ├─ 幻觉检测（NLI 模型 / LLM-as-Judge）
  ├─ 引用验证（答案中的事实 ↔ 源文档比对）
  ├─ 置信度评估
  └─ 不合格 → 回退重检索或降级返回
  ↓
[后处理层]
  ├─ 精确引用标注（文档名+章节+段落）
  ├─ 答案格式化（按场景调整输出格式）
  └─ 合规检查（敏感信息过滤）
  ↓
返回答案 + 引用 + 置信度 + 元数据
```

### 3.4 完整技术栈

#### 后端

| 组件 | 技术选型 | 与 L1 的变化 |
|------|---------|-------------|
| **语言** | Python 3.11+ | 不变 |
| **Web 框架** | FastAPI | 不变 |
| **Pipeline 编排** | LangGraph 或自研 StateGraph | **新增**：需要条件路由和状态管理 |
| **任务队列** | Celery + Redis（分优先级队列） | 升级：大/小文件分队列 |
| **关系数据库** | PostgreSQL 16 + pgvector 扩展 | 升级：可用 pgvector 做混合检索 |
| **向量数据库** | Qdrant 或 Milvus | 升级：需要更强的过滤和分片能力 |
| **全文检索** | Elasticsearch 或 PostgreSQL FTS | **新增**：BM25 稀疏检索 |
| **缓存** | Redis（答案缓存 + 语义缓存） | 升级：增加语义相似度缓存 |
| **知识图谱**（可选） | Neo4j 或 PostgreSQL + Apache AGE | **新增**：实体关系检索 |
| **可观测性** | LangSmith / LangFuse / Phoenix | **新增**：全链路追踪 |

#### 前端

| 组件 | 技术选型 | 与 L1 的变化 |
|------|---------|-------------|
| **框架** | React 18 / Next.js 14 + TypeScript | 可升级到 Next.js 做 SSR |
| **UI 库** | Shadcn/ui + Tailwind CSS | 更灵活的定制能力 |
| **图表** | ECharts 或 Recharts | **新增**：质量看板可视化 |
| **Markdown** | react-markdown + KaTeX + Mermaid | 升级：支持公式和图表渲染 |
| **引用组件** | 自研引用面板 | **新增**：点击引用跳转到原文高亮 |
| **管理后台** | 场景配置、Prompt 管理、质量看板 | **新增** |

#### 模型矩阵

| 环节 | 模型 | 规格 | 为什么需要 |
|------|------|------|-----------|
| **Embedding** | BGE-M3 或 Jina-Embeddings-v3 | 1024维 | 与 L1 相同，但需要支持多语言 |
| **稀疏 Embedding** | SPLADE++ 或 BGE-M3 稀疏向量 | - | **新增**：BM25 级别的精确匹配 |
| **Reranker** | BGE-Reranker-v2-M3 或 Cohere Rerank | Cross-Encoder | 与 L1 相同 |
| **NLI 模型** | DeBERTa-v3-large-mnli 或 bge-reranker-v2-gemma | - | **新增**：幻觉检测 |
| **意图分类** | 微调的 BERT/RoBERTa 或 LLM few-shot | 小模型即可 | **新增**：问题路由 |
| **生成 LLM（主力）** | DeepSeek-V3 / GPT-4o / Claude 3.5 Sonnet | 通过 API | 升级：需要更强的推理能力 |
| **生成 LLM（轻量）** | DeepSeek-V3-Lite / GPT-4o-mini / Qwen2.5-7B | API 或本地 | **新增**：辅助任务 |
| **评估 LLM** | GPT-4o-mini 或同级 | API | **新增**：LLM-as-Judge 质量评估 |

#### 算法与技术

| 环节 | 算法/技术 | 说明 |
|------|----------|------|
| **查询改写** | LLM Rewrite + 规则后处理 | 口语→检索语言转换 |
| **查询分解** | LLM Decomposition | 复杂问题拆成 2-3 个子问题 |
| **HyDE** | Hypothetical Document Embeddings | 先让 LLM 生成假设答案，用假设答案做检索 |
| **BM25** | Okapi BM25 | 经典稀疏检索，精确匹配能力强 |
| **混合检索融合** | RRF（Reciprocal Rank Fusion） | 多路检索结果的标准融合算法 |
| **上下文压缩** | LLM Extraction / Sentence Selection | 从长 chunk 中提取与问题最相关的句子 |
| **幻觉检测** | NLI（Natural Language Inference） | 判断"答案"是否被"上下文"蕴含 |
| **置信度评估** | 多信号加权：检索分数 + Rerank 分数 + NLI 分数 | 综合评估答案可靠性 |
| **语义缓存** | Embedding 相似度 + TTL | 语义相近的问题复用缓存答案 |
| **结构化分块** | Markdown Header Splitter / 正则提取章节 | 保留文档结构信息 |
| **元数据过滤** | 向量数据库 WHERE 条件 | 按文档类型、日期、标签过滤 |

### 3.5 L2 的局限性

- Pipeline 虽然有条件分支，但路由逻辑是预定义的，不能动态学习
- 多跳检索的跳数和策略是固定的
- 不能自主决定"要不要检索"或"用什么工具"
- 质量保障是事后检查，不是过程中的自适应调整
- 不能从用户反馈中自动优化自己

## 四、L3 自治级：Agentic RAG

### 4.1 能力定义

> 系统具备自主决策能力——能判断是否需要检索、用什么策略检索、答案是否合格、是否需要重来。能从历史交互中学习，持续优化自身表现。

### 4.2 与 L2 的本质区别

L2 是"按预设路线走，走不通就报告"，L3 是"自己规划路线，走不通就换路，走完了还会总结经验"。

核心差异在三个字：**自主性**。

1. **自主规划**：LLM 自己决定需要几步、每步做什么
2. **自主纠错**：发现答案有问题，自动回溯重做
3. **自主进化**：从反馈中学习，优化检索和生成策略

### 4.3 架构全景

```
用户问题
  ↓
[Agent 控制器] ← L3 的核心
  │
  ├─ 思考："这个问题需要什么信息？"
  │   ├─ 简单事实 → 直接检索
  │   ├─ 需要比较 → 分解为多个子问题
  │   ├─ 需要计算 → 调用计算工具
  │   ├─ 需要最新数据 → 调用外部 API
  │   └─ 已有足够信息 → 直接生成
  │
  ├─ 执行：调用工具
  │   ├─ Tool: 向量检索
  │   ├─ Tool: BM25 检索
  │   ├─ Tool: 知识图谱查询
  │   ├─ Tool: SQL 查询（结构化数据）
  │   ├─ Tool: 计算器
  │   ├─ Tool: 外部 API
  │   └─ Tool: 子 Agent（委派子任务）
  │
  ├─ 观察：评估工具返回的结果
  │   ├─ 信息充分 → 进入生成
  │   ├─ 信息不足 → 换策略重新检索
  │   └─ 信息矛盾 → 标记冲突，综合分析
  │
  ├─ 生成：基于收集的所有信息生成答案
  │
  ├─ 自检：验证答案质量
  │   ├─ 通过 → 返回
  │   ├─ 幻觉 → 删除幻觉部分，重新生成
  │   └─ 不完整 → 补充检索，增强答案
  │
  └─ 学习：记录本次交互
      ├─ 哪些检索策略有效
      ├─ 哪些 chunk 被引用
      └─ 用户反馈 → 更新策略权重
```

#### 算法与技术

| 环节 | 算法/技术 | 说明 |
|------|----------|------|
| **Agent 推理** | ReAct（Reasoning + Acting） | Agent 的核心推理框架：思考→行动→观察→循环 |
| **规划** | Plan-and-Execute / Tree of Thoughts | 复杂问题先规划步骤再执行 |
| **多跳检索** | Iterative Retrieval + Chain-of-Retrieval | 根据前一步结果决定下一步检索什么 |
| **知识图谱构建** | NER + Relation Extraction + Graph Embedding | 从文档自动构建知识图谱 |
| **图谱推理** | Cypher 查询 + Graph RAG | 基于图结构的关系推理 |
| **自适应检索** | Adaptive RAG（Self-RAG 变体） | LLM 自主决定是否需要检索、检索什么 |
| **自我反思** | Reflexion / Self-Refine | 生成答案后自我评估，不满意则修改 |
| **CRAG** | Corrective RAG | 检索结果不相关时，自动切换到其他数据源 |
| **幻觉检测** | NLI + Claim Decomposition + Fact Verification | 将答案分解为原子事实，逐一验证 |
| **学习优化** | 在线学习 / Bandit 算法 | 根据用户反馈动态调整检索策略权重 |
| **Prompt 优化** | DSPy / Auto-Prompt | 自动优化 Prompt 模板 |
| **评估** | RAGAS / DeepEval | 自动化 RAG 质量评估框架 |

## 五、三级对比总表

| 维度 | L1 基础级 | L2 专业级 | L3 自治级 |
|------|----------|----------|----------|
| **开发周期** | 4-8 周 | 12-20 周 | 24-40 周 |
| **团队规模** | 1-2 全栈 | 3-5 人（含 ML） | 6-10 人（含 ML + 数据） |
| **基础设施成本** | 低（单机可跑） | 中（需 GPU + 多服务） | 高（分布式 + 多 GPU） |
| **LLM API 成本** | 低（仅生成） | 中（生成+改写+评估） | 高（Agent 多轮调用） |
| **检索方式** | 向量 + 关键词 | 向量 + BM25 + 元数据 + RRF | 向量 + BM25 + 图谱 + 自适应 |
| **生成质量** | 依赖 LLM 能力 | 有质量保障兜底 | 自纠错 + 持续优化 |
| **幻觉控制** | 无 | NLI 检测 + 置信度 | 原子事实验证 + 自动修正 |
| **可扩展性** | 单知识库 | 多知识库 + 多场景 | 多知识库 + 多 Agent + 多数据源 |
| **学习能力** | 无 | 基于规则的策略切换 | 从反馈中自动优化 |

---

## 六、演进路径

```
L1 基础级
  │ 补齐：真 BM25 + 中文分块 + Token 分块
  │ 新增：幻觉检测 + 置信度评估
  │ 新增：意图分析 + 查询改写 + HyDE
  │ 新增：场景适配 + 离线评估
  ↓
L2 专业级
  │ 新增：Agent 控制器（ReAct 循环）
  │ 新增：工具调用能力
  │ 新增：知识图谱 + 多跳推理
  │ 新增：自适应学习
  ↓
L3 自治级
```

**核心建议**：不要跳级。先把 L2 做扎实（尤其是质量保障层），再考虑 L3。一个没有幻觉检测的 Agent 系统比没有 Agent 的系统更危险——因为它会"自信地犯错"。
