- **版本路线与能力等级**：V1 → V2.0 → V2.1 → V2.2 与 L1.8 → L2.5 → L2.7 → L2.9

便于产品、开发、测试一眼看清**整体进度**与**下一步工作**。

## 二、版本路线与能力等级

### 2.1 总体路线

| 版本   | 能力等级 | 定位                     | 状态     |
|--------|----------|--------------------------|----------|
| **V1** | **L1.8** | 基础级上限，触及 L2 门槛 | 已上线   |
| **V2.0** | **L2.5** | 检索质变 + 质量保障     | **进行中** |
| V2.1   | L2.7     | 智能与企业级（场景/权限/可观测） | 计划中   |
| V2.2   | L2.9     | 精细化与运营（缓存/Prompt/增量索引） | 计划中   |
| 远期   | L3       | 自治级（Agent + 自我学习） | 规划中   |

### 2.2 能力三级模型（L1 / L2 / L3）摘要

| 维度       | L1 基础级                 | L2 专业级                         | L3 自治级                 |
|------------|---------------------------|-----------------------------------|---------------------------|
| **核心特征** | 检索+生成，线性流水线     | 多路智能检索+质量保障+场景适配   | 自主决策+自我纠错+持续进化 |
| **类比**   | 会查资料的实习生         | 经验丰富的专业顾问                | 能自我学习的专家团队      |
| **错误处理** | 检索不到就说不知道       | 检索不到会换策略重试             | 能识别错误并自动修正      |
| **适用场景** | FAQ、内部知识查询        | 客服、培训、合规咨询              | 金融分析、法律推理、研究辅助 |

**演进建议**：不跳级。先把 L2 做扎实（尤其是质量保障层），再考虑 L3。

## 三、当前系统定位（V1 = L1.8）与对标

### 3.1 为何是 L1.8

- **L1 能力**：向量检索、关键词检索、Reranker、去重、查询扩展、多策略、流式输出、引用标注、检索日志、多轮对话等均已实现且较扎实。
- **未达 L2**：无真正的 BM25、无 RRF 多路融合、无幻觉检测/置信度/答案验证、无场景适配、无意图分析/HyDE/查询改写。
- **结论**：定为 **L1.8**（L1 上限，触及 L2 门槛）。

### 3.2 与开源项目对标

| 系统               | 定级   | 一句话                                       |
|--------------------|--------|----------------------------------------------|
| **Enterprise RAG** | **L1.8** | 检索链路完整且有工程深度，缺质量保障和自适应 |
| Langchain-Chatchat | L1.5   | 中文生态好，架构受框架限制                   |
| Haystack           | L1.7 框架 + L2.3 工具箱 | 框架线性，评估工具箱达 L2           |
| RAGFlow            | L2.2   | 文档理解+混合检索达 L2，缺智能查询层与质量保障层 |

## 四、V2.0 核心目标与范围（L1.8 → L2.5）

### 4.1 两大质变

1. **检索质量质变**  
   单路向量 → **BM25 + 向量混合 + RRF 融合 + 父文档检索 + 自适应深度 + 去噪**。

2. **答案可信度保障**  
   无质量验证 → **NLI 幻觉检测 + 置信度评估 + 答案验证 Pipeline + 智能拒答 + 引用验证 + 用户反馈（点赞/点踩）**。

**时间规划**：8 周（B1 Week 1–3 + B2 Week 4–6 + 集成验证）。

### 4.2 产品定位变化（V1 vs V2.0）

| 维度       | V1 (L1.8)           | V2.0 (L2.5)                          |
|------------|--------------------|--------------------------------------|
| 检索方式   | 向量 + 伪关键词     | BM25 + 向量混合 + RRF                |
| 分块能力   | 字符/句子           | 中文递归 / Token 级 / 父文档双层     |
| 答案可靠性 | 完全依赖 LLM       | NLI 幻觉检测 + 置信度评估            |
| 拒答能力   | 无                 | 系统级智能拒答                       |
| 引用准确性 | 后置余弦匹配       | NLI 验证引用准确性                   |
| 用户反馈   | 无                 | 点赞/点踩 + 反馈原因                 |

### 4.3 明确不在 V2.0 的功能（归入后续版本）

| 功能                                                                 | 计划版本 |
|----------------------------------------------------------------------|----------|
| 意图分析、查询改写、HyDE、查询分解                                   | V2.1     |

## 七、技术栈与关键实现

### 7.1 技术栈变更摘要

| 层级       | V1              | V2.0 |
|------------|-----------------|------|
| 关系数据库 | PostgreSQL 16   | PostgreSQL 16 + **pg_jieba** |
| NLI 模型   | 无              | **cross-encoder/nli-deberta-v3-base** |
| LLM        | DeepSeek/OpenAI | **Kimi K2.5 / K2.5-mini / Qwen2.5-7B**（三级架构） |
| 前端 SPA   | React + Vite + TS | React + Vite + **shadcn/ui + Tailwind CSS + Zustand** |
| 分词/Token | 无              | **jieba**、**tiktoken** |

### 7.2 数据库与反馈设计要点

- **chunks**：PostgreSQL 镜像表，含 `content_tsv`、`parent_chunk_id`、`section_title` 等；BM25 用 GIN(content_tsv)，向量仍在 ChromaDB。
- **retrieval_feedbacks**：V2.0 复用 V1 表，扩展 `rating`（thumbs_up/thumbs_down）、`reason`；不新建 qa_feedbacks 表。
- **retrieval_logs**：新增 `confidence_score`、`confidence_level`、`faithfulness_score`、`has_hallucination`、`retrieval_mode`、`refusal_reason`、`citation_accuracy`、`latency_breakdown`。

### 7.3 已完成的修复与配置要点

- **retrieval_log 模型**：已补充 `confidence_level`、`has_hallucination`、`retrieval_mode`、`citation_accuracy`、`latency_breakdown`，与迁移 V2.0_006 一致。
- **Kimi Provider**：`app/llm/kimi.py` 已存在，`build_chat_provider(provider="kimi")` 支持，temperature 固定 1.0。
- **.env**：V2.0 版本说明完整，包含检索、质量保障、LLM 及可选 `LLM_TASK_OVERRIDES`（如 query_expansion 用不同模型）。
- **Feedback API**：实际路径为 `POST /api/retrieval/feedback`（建议 PRD/TRD 统一描述为该路径）。
- **pg_jieba**：使用预构建镜像 `sdrzlyz/pg-ts-jieba:16`，无需自建 Dockerfile。

## 九、总体结论与建议

### 9.1 设计完备度

- **14/20 模块**：TRD 覆盖完备（基础设施 3、检索引擎 6、质量保障 4、前端基础 1）。
- **6/20 模块**：存在小缺口（R-4 向量检索器接口统一、V-5 反馈与 V1 迁移说明、F-2 布局清单、F-3 类型表、F-4 QASettingsSheet 联动）。
- **横切**：建议优先在 TRD 中补充 **pipeline / qa_service 串联** 的伪代码或流程图，再开发/联调；其次补 document_tasks、反馈 API 与 V1 表关系、kimi 设计说明。

### 9.2 实现进度

- 骨架、迁移、核心 API、Kimi、.env、retrieval_log 与反馈表扩展均已完成。
- **待实现**：前端 **QASettingsSheet** 根据策略的 `expansion_enabled` 禁用「查询扩展模式」并展示提示文案（逻辑见 `docs/V2.0_检索与配置汇总.md` 第四节）。

### 9.3 建议执行顺序

1. **高优**：在 TRD 中补充 pipeline/qa_service 端到端串联说明（伪代码或流程图）。
2. **中优**：TRD 中补全 V-5 反馈 API 路径与 V1 表迁移说明；实现 QASettingsSheet 策略与查询扩展联动。
3. **低优**：TRD 中补充 R-4 向量检索器接口约定、F-2 布局清单、F-3 类型表、kimi_provider 小节；document_tasks 串联一节。

### 9.4 关键结论（总览）

1. **版本路线清晰**：V1(L1.8) → V2.0(L2.5) → V2.1(L2.7) → V2.2(L2.9)，不跳级。
2. **V2.0 范围明确**：检索质变（混合+父文档+自适应+去噪）+ 质量保障（NLI+置信度+拒答+引用验证+反馈）；意图/场景/权限/评估/LangFuse 等在 V2.1。
3. **TRD 已覆盖 20 模块**：四大领域在 TRD 中均有对应设计（含 pipeline/qa_service 串联、反馈迁移、Kimi、document_tasks 等）。
4. **实现进度高**：骨架、迁移、核心 API、Kimi、.env、retrieval_log 与反馈表扩展均已完成；仅前端 QASettingsSheet 与策略联动待完善。
5. **反馈与 API 以实现为准**：反馈使用 `POST /api/retrieval/feedback`，表为 `retrieval_feedbacks` 扩展，PRD/TRD 已与实现对齐。
