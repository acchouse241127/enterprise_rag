# 知识库使用说明

本文档说明知识库的完整配置体系：**分块与上下文**（分块模式、父文档检索模式、扩展相邻 chunk 数）与**检索与查询**（检索模式、检索策略、TopK、查询扩展模式），以及在不同业务场景下的配置建议与调优方法。

---

## 一、分块模式

分块模式决定文档如何被切分成检索与召回的文本块（chunk）。不同模式适用于不同文档类型与检索需求。

### 1.1 中文递归分块（推荐）

| 项目 | 说明 |
|------|------|
| **描述** | 按中文语义边界递归切分，优先保持语义完整性。 |
| **适用场景** | 大多数以中文为主的知识库；希望分块能保持上下文连贯、提高检索准确性与相关性时。 |

### 1.2 Token 级分块

| 项目 | 说明 |
|------|------|
| **描述** | 按 Token 数控制块大小，贴合 LLM 上下文窗口。 |
| **适用场景** | 需精确控制送入 LLM 的文本长度、优化处理效率、避免超出上下文限制时；如问答系统中希望每个检索块都能完整放入模型输入。 |

### 1.3 句子边界分块

| 项目 | 说明 |
|------|------|
| **描述** | 按句号、问号、感叹号等句子结束符及段落分隔符切分。 |
| **适用场景** | 内容结构清晰、以句子和段落为单位的文本：法律文档、正式报告、新闻等；需确保每块为完整句子或段落、避免截断导致信息丢失时。 |

### 1.4 字符滑动窗口（V1 兼容模式）

| 项目 | 说明 |
|------|------|
| **描述** | 固定大小滑动窗口按字符数切分，与旧版 V1 行为一致。 |
| **适用场景** | 对分块方式要求简单统一，或需与 V1 系统/工作流兼容；非结构化、格式不一致的文本做粗粒度分块时。 |

### 1.5 分块模式选择小结

| 需求 | 建议模式 |
|------|----------|
| 新建知识库、无特殊需求 | **中文递归分块** |
| 需兼容旧版或固定长度 | **字符滑动窗口** |
| 内容以完整句/段为主 | **句子边界分块** |
| 需精确控制块大小以适配 LLM | **Token 级分块** |

## 二、父文档检索模式

父文档检索在「先按小块检索」的基础上，将命中小块所属的更大范围（父文档或相邻块）作为完整上下文提供给 LLM。

### 2.1 动态扩展（推荐，无需重新索引）

| 项目 | 说明 |
|------|------|
| **描述** | 检索时动态扩展上下文，不依赖预先建立的物理双层索引，无需重新索引。 |
| **适用场景** | 追求效率与灵活性、文档更新频繁、希望更新尽快体现于检索、资源敏感需减少重索引成本时；多数场景的默认推荐。 |

### 2.2 物理双层（精确，需重新索引）

| 项目 | 说明 |
|------|------|
| **描述** | 预先建立子块与父文档两层索引，检索更精确，但需在文档或结构变更后重新索引。 |
| **适用场景** | 对检索准确性与上下文完整性要求极高（如法律、医疗、合规）；文档相对稳定、可接受重索引；需严格控制父文档边界时。 |

### 2.3 关闭（V1 兼容模式）

| 项目 | 说明 |
|------|------|
| **描述** | 不启用父文档检索，仅按单层块检索，行为与 V1 一致。 |
| **适用场景** | 从 V1 迁移或与 V1 工作流兼容、仅需块级检索、排查父文档相关问题时；或历史/仅针对 V1 调优的知识库。 |

### 2.4 父文档检索模式选择小结

| 需求 | 建议模式 |
|------|----------|
| 新建库、文档常更新、不想频繁重索引 | **动态扩展** |
| 法律/医疗等需精确父文档、文档较稳定 | **物理双层** |
| 兼容 V1 或不需要父文档上下文 | **关闭** |

## 三、扩展相邻 chunk 数

该配置在「检索命中某个 chunk」之后，以该块为中心，向前、向后各取 N 个相邻 chunk，与命中块一起组成送给 LLM 的上下文。

### 3.1 作用说明

- **含义**：N 为「扩展相邻 chunk 数」。例如 N=2 时，最终上下文 = 前 2 个 + 命中块 + 后 2 个，共 5 个 chunk。
- **目的**：在块较小时增加上下文连贯性，减少因单块信息不足导致的回答不完整或理解偏差；可与父文档检索配合，在命中块周围做精细化扩展。

### 3.2 取值对效果的影响

| 取值 | 效果 | 优点 | 缺点 |
|------|------|------|------|
| **较大（3、4 或更多）** | 上下文更宽 | 信息更全、减少遗漏 | Token 消耗大、可能引入噪音、成本上升 |
| **较小（0、1）** | 上下文更聚焦 | Token 少、噪音低 | 可能丢失重要背景、回答不完整 |
| **适中（2）** | 平衡 | 多数场景下兼顾完整性与成本 | 需结合 chunk 大小与文档类型微调 |

### 3.3 使用与调参建议

- **起步**：无把握时从 **1 或 2** 开始。
- **回答不完整、缺上下文**：可尝试增至 3 或 4。
- **回答啰嗦、无关内容多或成本偏高**：可尝试 1 或 0。
- **与分块、父文档联动**：chunk 越小，可适当增大扩展数；已启用父文档且父文档较大时，扩展数可适当减小。
- **原则**：在「上下文完整性」与「Token 消耗/噪音」之间权衡，按实际效果迭代。

## 四、检索模式

检索模式决定**用什么算法从知识库中找文档**：纯关键词、纯语义，或两者结合。

### 4.1 混合检索（推荐）

| 项目 | 说明 |
|------|------|
| **功能** | BM25（关键词全文检索）+ 向量检索（语义相似度）同时进行，再用 **RRF（Reciprocal Rank Fusion）** 融合两份结果，得到最终排序。 |
| **特点** | 既保留「精确词匹配」，又保留「同义、换说法」的语义匹配，一般召回更全、排序更稳。 |
| **适用场景** | 通用场景、不确定用户会怎么问时；既可能用原文关键词，也可能用口语/同义表达；希望兼顾准确率和召回率时。**多数知识库推荐默认使用。** |

### 4.2 向量检索

| 项目 | 说明 |
|------|------|
| **功能** | 仅用**语义相似度**：把 query 和文档块都变成向量，按向量距离/相似度排序，不看字面是否一致。 |
| **特点** | 能理解「意思相近但用词不同」；不依赖关键词，适合概念式、换说法、换问法。 |
| **适用场景** | 用户常问概念、同义表达、换一种说法；文档专业术语多、用户用口语问；更看重「意图匹配」而非「词匹配」时。 |

### 4.3 BM25 检索

| 项目 | 说明 |
|------|------|
| **功能** | 纯**关键词全文检索**：按 query 中的词在文档中的出现频率、分布做 BM25 打分排序，不做语义理解。 |
| **特点** | 专有名词、编号、法规条款号、型号等「必须字面匹配」时表现好；实现简单、延迟可控。 |
| **适用场景** | 用户已知准确关键词（如条款号、型号、标准名）；技术/法律/规范类文档，用语规范、一字不能差；或对语义不敏感、主要做「关键词查找」时。 |

### 4.4 检索模式选择小结

| 需求 | 建议模式 |
|------|----------|
| 不确定选哪个、通用场景 | **混合检索** |
| 强调概念、同义、换说法 | **向量检索** |
| 强调精确词、编号、规范用语 | **BM25 检索** |

## 五、检索策略

检索策略决定**在「召回多少、多准、多快」之间的侧重**，影响检索阶段的参数组合。

### 5.1 default（默认）

| 项目 | 说明 |
|------|------|
| **功能** | 在**召回率**和**响应速度**之间做平衡：既尽量多找到相关结果，又控制耗时。 |
| **适用场景** | 日常问答、没有特别偏好的场景；既要结果够全，又要响应可接受时。**无明确需求时选这个。** |

### 5.2 high_recall（高召回）

| 项目 | 说明 |
|------|------|
| **功能** | 优先**多返回结果**，放宽一些精度要求，避免漏掉可能相关的内容。 |
| **适用场景** | 探索式、调研式查询；用户不确定关键词、想全面了解；头脑风暴、法规/竞品梳理等。可接受结果里有一些不太相关的，但不能漏。 |

### 5.3 high_precision（高精度）

| 项目 | 说明 |
|------|------|
| **功能** | 优先**只返回最相关的结果**，减少无关文档，降低噪音。 |
| **适用场景** | 问题非常具体（查某一条、某一概念定义）；用户要精准答案、不想在一堆结果里筛选；事实核查、定义查询、条款精确定位等。 |

### 5.4 low_latency（低延迟）

| 项目 | 说明 |
|------|------|
| **功能** | 优先**最快响应**，可能减少检索量或简化排序，换取延迟降低。 |
| **适用场景** | 实时对话、在线客服、搜索建议等对首字/首条延迟敏感的场景；监控大屏、实时看板等。可接受在召回量/精度上略做牺牲。 |

### 5.5 检索策略选择小结

| 需求 | 建议策略 |
|------|----------|
| 一般用 | **default** |
| 要尽量不漏 | **high_recall** |
| 要尽量精准、少噪音 | **high_precision** |
| 要尽量快 | **low_latency** |

## 六、TopK（检索数量）

| 项目 | 说明 |
|------|------|
| **功能** | 控制**最终参与排序并送入后续环节（如重排/上下文组装）的 chunk 数量上限**。例如 TopK=5 表示最多取 5 个最相关的块。 |
| **可选值** | 常见为 3、5、10、15、20 等；值越大，候选越多、上下文越丰富，但 token 与噪音也会增加。 |

### 6.1 取值与场景

| 取值 | 特点 | 适用场景 |
|------|------|----------|
| **3** | 上下文最少、最聚焦，token 省 | 问题非常具体、只需一两处依据；或成本/延迟敏感、模型上下文很紧时。 |
| **5** | 平衡 | **默认推荐**：多数单轮问答够用，既不太吵也不太缺。 |
| **10** | 上下文更足 | 问题较开放、需要多角度依据；或文档 chunk 较小、单块信息少时。 |
| **15 / 20** | 上下文最多 | 综合型、综述型问题；多文档对比；或 chunk 很小、必须「多块拼一段」时。注意 token 与噪音。 |

**注意**：TopK 是检索阶段的数量；若还开启了父文档检索、扩展相邻 chunk，最终送给 LLM 的上下文会在此基础上再扩展，需一起考虑 token 与效果。

## 七、查询扩展模式

查询扩展在**用户输入之后、正式检索之前**，对 query 做改写或扩展（加同义词、拆问法、补全意图等），再用扩展后的 query 去检索，提高召回。

### 7.1 规则版

| 项目 | 说明 |
|------|------|
| **功能** | **基于规则的查询扩展**：用预置规则（如同义词表、领域词典、关键词映射）对 query 做扩展或替换，不加 LLM 调用。 |
| **特点** | 延迟低、成本低、行为可预期、易控；扩展能力受规则覆盖范围限制。 |
| **适用场景** | 领域术语固定、同义表达可枚举（如制度、合同、产品手册）；对延迟/成本敏感；或不允许/不方便用 LLM 做扩展时。 |

### 7.2 LLM 版

| 项目 | 说明 |
|------|------|
| **功能** | **基于 LLM 的查询扩展**：用大模型理解用户意图，生成改写句、同义问法、子问题等，再用这些扩展 query 去检索。 |
| **特点** | 能处理口语化、模糊问法、多意图，扩展更灵活；多一次 LLM 调用，延迟和成本更高。 |
| **适用场景** | 用户问法多样、口语多、问题较长或含糊；需要「意图理解」再检索；对首轮检索召回要求高、可接受多几百毫秒和一定成本时。 |

### 7.3 混合版

| 项目 | 说明 |
|------|------|
| **功能** | **规则 + LLM 结合**：先规则做基础扩展（如术语、同义词），再用 LLM 做意图改写或补充问法，两者结果一起用于检索。 |
| **特点** | 规则保证稳定、可控，LLM 补足灵活度；成本与延迟介于规则版和纯 LLM 版之间。 |
| **适用场景** | 既有固定术语/规范表达，又有口语化/多说法；既要可控又要更好召回时。适合多数企业知识库的进阶配置。 |

### 7.4 查询扩展模式选择小结

| 需求 | 建议模式 |
|------|----------|
| 术语固定、要省成本、要低延迟 | **规则版** |
| 问法多样、要意图理解、可接受多一次 LLM | **LLM 版** |
| 两者都要 | **混合版** |

### 8.2 分场景说明（含配置理由）

#### 场景 1：企业制度/规章检索  
**分块：中文递归 | 父文档：动态扩展 | 扩展：2 | 检索：混合 | 策略：default | TopK：5 | 查询扩展：规则版**

- 制度多为中文、按条/款/项组织，递归分块保持语义完整；制度常修订，动态扩展免重索引；扩展 2 便于带出前后条款。
- 用户既可能用原文表述也可能口语化，混合检索兼顾关键词与语义；default 平衡召回与速度；TopK 5 足够单轮问答；制度术语相对固定，规则版扩展即可。

#### 场景 2：法律合同条款问答  
**分块：句子边界 | 父文档：物理双层 | 扩展：1 | 检索：混合 | 策略：high_precision | TopK：3～5 | 查询扩展：规则版**

- 法律语句不能截断，句子边界 + 物理双层保证条款归属精确；扩展 1 控制噪音。
- 需精确命中条款，high_precision 减少无关结果；TopK 不宜大；法律用语规范，规则版扩展。

#### 场景 3：医疗/合规文档解读  
**分块：句子边界 | 父文档：物理双层 | 扩展：2 | 检索：混合 | 策略：high_precision | TopK：5 | 查询扩展：规则版**

- 与法律类似，句子边界 + 物理双层便于审计；扩展 2 适应「定义 + 适用情形 + 例外」等跨段内容。
- 高精度优先；专业术语多，规则版可控。

#### 场景 4：产品手册/FAQ  
**分块：中文递归 | 父文档：动态扩展 | 扩展：0～1 | 检索：混合 | 策略：default | TopK：5 | 查询扩展：规则版**

- 单条 FAQ 或单节手册常可独立回答，扩展 0～1 控制 token；手册更新频繁，动态扩展。
- 通用平衡配置；产品术语可预置，规则版低成本。

#### 场景 5：技术/API 文档  
**分块：中文递归 或 Token 级 | 父文档：动态扩展 | 扩展：1～2 | 检索：混合 | 策略：default | TopK：5～10 | 查询扩展：混合版**

- 结构清晰、常更新；代码块多时可用 Token 级；扩展 1～2 带出参数与示例。
- 用户问法既有术语也有口语，混合检索 + 混合版扩展兼顾；TopK 略高以覆盖多节内容。

#### 场景 6：财报/研报分析  
**分块：Token 级 | 父文档：动态扩展 | 扩展：1 | 检索：混合 | 策略：default | TopK：5 | 查询扩展：规则版**

- 大段表格与长段落，Token 级对齐 LLM 窗口；扩展 1 补充上下文即可。
- 数字与术语并存，混合检索；规则版控制成本。

#### 场景 7：新闻/舆情摘要  
**分块：句子边界 | 父文档：动态扩展 | 扩展：2 | 检索：混合 | 策略：high_recall | TopK：10 | 查询扩展：LLM 版**

- 按句按段、内容常更新；扩展 2 保证事件前后连贯。
- 探索式查询多，high_recall + TopK 10 避免漏报；问法多样，LLM 版扩展提升召回。

#### 场景 8：多语言或混合文档  
**分块：Token 级 | 父文档：动态扩展 | 扩展：2 | 检索：混合 | 策略：default | TopK：10 | 查询扩展：混合版**

- 中英混排时 Token 级更一致；扩展 2 弥补多语言语义边界不规整。
- 混合检索与混合版兼顾多语言表述与术语。

#### 场景 9：V1 迁移/历史知识库  
**分块：字符滑动窗口 | 父文档：关闭 | 扩展：0 | 检索：混合 | 策略：default | TopK：5 | 查询扩展：规则版**

- 与 V1 行为一致，不做父文档与扩展；分块用字符滑动窗口兼容旧索引。
- 检索侧保持通用配置，便于对比与回滚。

#### 场景 10：高并发/成本敏感  
**分块：中文递归 | 父文档：动态扩展 或 关闭 | 扩展：0～1 | 检索：混合 | 策略：low_latency | TopK：3 | 查询扩展：规则版**

- 少 token、少扩展，优先响应与成本；可选关闭父文档进一步减负。
- low_latency + TopK 3 + 规则版，全部围绕低延迟与低成本。

#### 场景 11：客服工单知识库  
**分块：中文递归 | 父文档：动态扩展 | 扩展：1 | 检索：混合 | 策略：low_latency | TopK：5 | 查询扩展：LLM 版**

- 需快速响应，扩展 1、动态扩展；用户描述问题口语化、多样，LLM 版扩展提高命中。
- low_latency 保证首条延迟可接受。

#### 场景 12：招投标文档检索  
**分块：句子边界 | 父文档：物理双层 | 扩展：1 | 检索：混合 | 策略：high_precision | TopK：5 | 查询扩展：规则版**

- 标书条款边界清晰、归属需精确，句子边界 + 物理双层；扩展 1 减少无关条款。
- 常按条款号或关键词查，high_precision + 规则版。

#### 场景 13：内部培训材料问答  
**分块：中文递归 | 父文档：动态扩展 | 扩展：2 | 检索：向量检索 | 策略：high_recall | TopK：10 | 查询扩展：混合版**

- 概念多、问法多样，递归分块 + 扩展 2 保证概念解释完整。
- 强调「意思相近」而非字面匹配，向量检索 + high_recall；混合版兼顾术语与口语。

#### 场景 14：政策解读与比对

#### 场景 15：专利/技术交底书检索

- 技术术语精确 + 语义（同族、相近方案），Token 级控制块长；扩展 1～2 带出技术上下文。

## 九、使用与调优顺序建议

1. **先按场景选配置**：根据业务类型在「八、20 个真实场景」总览表中选择对应的完整配置（分块、父文档、扩展数、检索模式、检索策略、TopK、查询扩展）作为起点。
2. **按效果微调**：  
   - 回答缺上下文 → 保持或启用父文档，或适当增大扩展数、TopK。  
   - 回答啰嗦、无关多、成本高 → 减小扩展数、TopK，或选用 high_precision、规则版查询扩展。  
   - 漏检多 → 可试 high_recall、增大 TopK、或 LLM/混合版查询扩展。  
   - 延迟高 → 可试 low_latency、减小 TopK、规则版查询扩展。
3. **分块、上下文、检索一起看**：chunk 越小，往往需要略大的扩展数或更强的父文档；chunk 越大，扩展数可略小。检索侧：要准选 high_precision + 小 TopK，要全选 high_recall + 大 TopK，要快选 low_latency + 小 TopK + 规则版。
4. **无固定最佳值**：所有配置均需结合文档内容、chunk 大小、查询类型和 LLM 上下文窗口做实验与调优。

## 十、配置项速查

| 配置项 | 可选值/范围 | 主要作用 |
|--------|-------------|----------|
| **分块模式** | 中文递归分块、Token 级分块、句子边界分块、字符滑动窗口 | 决定文档如何切块，影响检索粒度与语义完整性 |
| **父文档检索模式** | 动态扩展、物理双层、关闭 | 决定是否以及如何提供「命中块所属的更大上下文」 |
| **扩展相邻 chunk 数** | 0, 1, 2, 3, … | 命中块前后各取 N 个相邻块，组成更完整的上下文 |
| **检索模式** | 混合检索、向量检索、BM25 检索 | 决定用关键词、语义或两者融合的方式检索 |
| **检索策略** | default、high_recall、high_precision、low_latency | 在召回量、精度、响应速度之间的侧重 |
| **TopK（检索数量）** | 3, 5, 10, 15, 20, … | 参与后续环节的 chunk 数量上限 |
| **查询扩展模式** | 规则版、LLM 版、混合版 | 检索前是否以及如何扩展/改写用户 query |
