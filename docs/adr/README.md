# ADR-001: 向量数据库选型 - ChromaDB

## 状态

已采纳

## 背景

Enterprise RAG 系统需要一个向量数据库来存储文档的 Embedding 并支持相似度检索。

## 决策

选择 **ChromaDB** 作为向量数据库。

## 理由

1. **轻量级部署**：单进程运行，无需独立集群
2. **原生支持 Python**：与后端技术栈（FastAPI）完美匹配
3. **内置持久化**：支持本地文件存储
4. **多语言 Embedding 支持**：BGE-M3 模型中英文效果俱佳
5. **简单 API**：快速开发，降低学习成本

## 替代方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| Milvus | 高性能、分布式 | 部署复杂，需要独立集群 |
| Pinecone | 托管服务、免运维 | 商业服务，数据外流 |
| Weaviate | GraphQL 支持 | 资源占用较大 |
| pgvector | 无需新增组件 | 性能不如专用向量库 |

## 影响

- 向量检索通过 HTTP 调用 ChromaDB（默认端口 8001）
- 需要管理 Collection 的生命周期
- 未来可通过抽象层迁移到其他向量库

# ADR-002: 消息队列选型 - Redis + Celery

## 状态

已采纳

## 背景

文档解析和向量化是计算密集型任务，需要异步处理以避免阻塞 API 请求。

## 决策

选择 **Redis + Celery** 作为消息队列和任务调度方案。

## 理由

1. **Redis 已有部署**：作为缓存和 Session 存储，复用基础设施
2. **Celery 成熟稳定**：Python 生态最流行的任务队列
3. **支持任务重试**：失败自动重试，提高可靠性
4. **Worker 水平扩展**：可通过增加 Worker 实例提升吞吐量
5. **任务状态追踪**：支持查询任务执行进度

## 替代方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| RabbitMQ | 功能强大、稳定 | 部署复杂，需要独立运维 |
| AWS SQS | 托管服务 | 依赖云平台 |
| Django-Q | 轻量 | 功能有限 |

## 影响

- 需要启动 Celery Worker 进程
- 任务执行状态通过 Redis 存储
- 支持任务优先级和路由

## 日期

# ADR-003: 前端技术选型 - React SPA

## 状态

已采纳

## 背景

系统需要一个现代化的 Web 前端，支持流式响应、引用溯源等复杂交互。

## 决策

选择 **React + Vite + TypeScript** 构建 SPA（单页应用），作为主前端。

## 理由

1. **React 生态成熟**：组件化开发，状态管理丰富
2. **Vite 极速构建**：开发体验好，HMR 快
3. **TypeScript 类型安全**：减少运行时错误
4. **SSE 原生支持**：流式问答体验
5. **SPA 体验流畅**：无页面刷新，用户体验好

## 替代方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| Streamlit | 快速开发 | 定制性差，体验一般 |
| Vue.js | 渐进式、易学 | 团队 React 经验更多 |
| Next.js | SSR 支持 | 增加部署复杂度 |

## 影响

- 前端独立构建和部署
- 通过 Nginx 代理后端 API
- 保留 Streamlit 作为内部调试工具

## 日期

# ADR-004: Embedding 模型选型 - BGE-M3

## 状态

已采纳

## 背景

需要选择一个支持中英文、效果好的 Embedding 模型用于文档向量化。

## 决策

选择 **BAAI/bge-m3** 作为 Embedding 模型。

## 理由

1. **多语言支持**：中英文效果俱佳
2. **开源免费**：无 API 调用成本
3. **本地部署**：数据不外流
4. **效果领先**：MTEB 榜单排名靠前
5. **灵活性**：支持多种检索模式

## 替代方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| OpenAI Embedding | 效果稳定 | 商业收费，数据外流 |
| text-embedding-ada-002 | 便宜 | 中文效果一般 |
| M3E | 中文优化 | 英文效果不如 BGE |

## 影响

- 需要本地 GPU/CPU 资源运行模型
- 首次加载需要下载模型权重
- 可配置远程 Embedding 服务

## 日期

# ADR-005: 检索策略 - 多路召回 + Reranker

## 状态

已采纳

## 背景

单一向量检索可能遗漏重要结果，需要更鲁棒的检索策略。

## 决策

采用 **多路召回（向量 + 关键词）+ Reranker 重排序** 的检索策略。

## 理由

1. **向量检索**：语义相似度匹配
2. **关键词检索**：精确匹配，补充向量遗漏
3. **Reranker**：精排提升相关性
4. **可配置策略**：high_recall/high_precision/low_latency
5. **去重机制**：SimHash 去重避免冗余

## 影响

- 检索延迟增加（Reranker 调用）
- 可通过配置开关各路召回
- API 支持 strategy 参数动态选择

## 日期

2026-02-21
