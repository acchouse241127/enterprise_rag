## 1. 你的核心目标（最终一致版本）

- 建立规范开发流程，并完成后端、Streamlit、SPA 三端能力对齐。
- 把 Query Expansion 做成可配置能力：
  - 规则版（rule）

## 3. 当前真实状态（基于代码现状核对）

## 3.1 Query Expansion 请求级控制仍未打通

- 当前 `/qa/ask` 与 `/qa/stream` 请求体只接收 `strategy`，未接收请求级 `query_expansion_mode` 或请求级 LLM 配置。
- 现状是“全局配置驱动”：
  - `RETRIEVAL_QUERY_EXPANSION_MODE` 从 `.env` / `settings` 生效
  - `LLM_TASK_OVERRIDES.query_expansion` 可指定扩展用模型

### 3.2 hybrid 目前未满足“rule + llm 真组合”

- `hybrid` 现实现偏保守，核心路径仍是 rule 优先且未充分合并 LLM 扩展结果，与你最新目标不一致。

### 3.3 前端目前缺失“云端/本地 + 完整模型表单”

- SPA 与 Streamlit 当前只提供 strategy 维度，未提供 query expansion 的模式切换与模型参数输入。

## 4. 已确认并冻结的新目标（下一轮必须执行）

执行顺序按你最新要求固定为：

1. 先升级 `hybrid` 为真正 `rule + llm` 组合。  
2. 再做前后端改造：支持“云端/本地可选 + 完整模型信息可填”。  
3. 最后统一执行测试与验证（后端、前端、集成、Docker）。

## 5. 目标设计（建议落地形态）

### 5.1 前端交互目标

- Query Expansion 模式选择：
  - 规则版（rule）
  - LLM 云端版（展示模型名，例如 `kimi-k2.5`）
  - LLM 本地版（当前未部署时显示占位文案：`本地LLM（未部署）`）
  - hybrid（可选展示为“混合（规则+LLM）”）
- 模型配置区（完整）：
  - `provider`
  - `model_name`
  - `base_url`
  - `api_key`
  - `temperature`
  - `timeout_seconds`
  - `max_retries`
  - `retry_base_delay`

### 5.2 后端接口目标

- 在 QA 请求体中增加扩展字段（ask/stream 均支持）：
  - `query_expansion_mode`
  - `query_expansion_target`（`cloud | local | default`）
  - `query_expansion_llm`（对象，承载完整模型配置）
- 新增元信息接口（推荐）：
  - 提供默认模型名、local 可用性、可选 provider 列表，供前端渲染提示。

### 5.3 混合模式目标（关键）

- `hybrid` 执行顺序建议：
  - 先运行 rule 生成候选 query
  - 再运行 llm 生成候选 query
  - 去重合并（保持原 query 最高优先级）
  - 按上限截断（例如 `1 + max_extra`）

## 6. 配置现状（.env 已有基础）

当前 `.env` 已具备：

- `RETRIEVAL_QUERY_EXPANSION_MODE`
- `RETRIEVAL_QUERY_EXPANSION_ENABLED`
- `LLM_TASK_OVERRIDES`（可为 `query_expansion` 指定云端或本地模型）

说明：

- 在请求级改造完成前，运行时仍由这些全局配置主导。
- 已发现 `.env` 中存在明文 API Key，后续建议轮换并迁移到更安全的密钥管理方式。

## 8. 风险与注意事项

- 安全：明文 API Key 风险较高，建议立即轮换。
- 兼容性：新增请求字段需保持“可选”，避免破坏旧调用。
- 性能：hybrid 真实组合会增加延迟，应设置合理 `max_extra` 并做去重截断。
- 体验：本地 LLM 未部署时前端需明确提示，不应导致提问失败。
