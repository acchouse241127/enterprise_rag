# V2.0 前端功能迭代说明：检索看板与文件夹同步

本文档汇总 V2.0 阶段对 React SPA 前端的检索质量看板迁移、文件夹同步功能完善及 Docker 部署友好化方案，便于产品、开发与运维查阅。

---

## 一、检索质量看板迁移（Streamlit → SPA）

### 1.1 背景与目标

- **来源**：Streamlit 页面 `frontend/pages/7_检索质量看板.py`
- **目标**：在 React SPA（`frontend_spa`）中实现同等功能，为淘汰 Streamlit 前端做准备
- **参考**：`docs/V2.0_前端与运维汇总.md` 第四节

### 1.2 已实现功能

| 模块 | 说明 |
|------|------|
| **筛选条件** | 知识库下拉、时间范围（7/14/30 天）、刷新按钮 |
| **核心指标卡片** | 总查询数、平均最高分、平均响应时间、无用反馈率、有用/无用反馈数、问题样本数、平均返回块数（共 8 项） |
| **查询趋势** | 每日查询量柱状图、每日平均分数折线图（recharts） |
| **检索日志列表** | 分页、反馈状态筛选（全部/有反馈/无反馈/有用/无用）、单条👍/👎/🔖 操作 |
| **按知识库统计** | 表格展示各知识库的查询数、平均分、平均耗时、有用/无用反馈数 |
| **问题样本列表** | 已标记问题样本的检索记录，支持刷新 |
| **无数据与错误** | 检索日志未启用提示、无数据排查步骤、错误信息展示 |

### 1.3 涉及文件

- **页面**：`frontend_spa/src/pages/Dashboard.tsx`
- **API**：`frontend_spa/src/api/dashboard.ts`（沿用既有接口）
- **类型**：`frontend_spa/src/api/types.ts`（RetrievalStats、DailyStats、KbStats、RetrievalLog 等）
- **测试**：`frontend_spa/src/pages/Dashboard.test.tsx`（Vitest + Testing Library）
- **依赖**：`recharts` 用于趋势图表

### 1.4 后端接口（无变更）

- `GET /api/retrieval/stats`、`GET /api/retrieval/stats/by-date`、`GET /api/retrieval/stats/by-knowledge-base`
- `GET /api/retrieval/logs`、`GET /api/retrieval/logs/{log_id}`
- `POST /api/retrieval/feedback`、`POST /api/retrieval/feedback/{feedback_id}/mark-sample`
- `GET /api/retrieval/samples`

### 1.5 测试与验证

- 脚本：`npm run test:run`、`npm run test:coverage`
- 用例：Dashboard 标题、筛选区、指标卡片、趋势区、日志区、分页、反馈操作、无数据/未启用提示等
- 构建：`npm run build` 通过

## 二、文件夹同步功能完善

### 2.1 问题与目标

- **原状**：SPA 文件夹同步页缺少知识库选择、未拉取/展示后端配置、无删除能力，且监控路径仅能手动输入
- **目标**：功能对齐 Streamlit 的 `6_文件夹同步.py`，并支持用户在前端「选择本地文件夹」

### 2.2 已实现内容

| 项目 | 说明 |
|------|------|
| **知识库选择** | 下拉选择要配置同步的知识库 |
| **配置拉取与展示** | 选择知识库后自动拉取并展示当前同步配置（目录、间隔、文件模式、启用状态） |
| **创建/编辑/删除** | 无配置时显示创建表单；有配置时可修改或删除 |
| **本地文件夹选择** | 使用浏览器 File System Access API（`showDirectoryPicker`），点击「选择文件夹」可浏览本地任意目录，选中的文件夹名称会填入「同步目录」输入框 |
| **同步日志** | 展示该知识库的同步历史（状态、耗时、扫描/新增/更新/删除数） |
| **立即同步** | 手动触发一次同步 |

### 2.3 类型与 API 调整

- **类型**（`frontend_spa/src/api/types.ts`）：  
  `FolderSyncConfig`、`FolderSyncLog`、`FolderSyncConfigResponse`、`SyncNowResponse` 等与后端一致；新增 `DeleteSyncConfigResponse`
- **API**（`frontend_spa/src/api/knowledge-base.ts`）：  
  `getFolderSyncConfig`、`updateFolderSyncConfig`（请求体使用 `directory_path`）、`deleteFolderSyncConfig`、`syncFolderNow`、`getFolderSyncLogs(kbId, limit?)`

### 2.4 浏览器兼容与说明

- **文件夹选择器**：依赖 `showDirectoryPicker`，Chrome 86+、Edge 86+ 等支持；不支持的浏览器仅显示输入框并提示「请手动输入路径」
- **路径含义**：同步目录为**服务端（或容器内）路径**；Docker 部署时需通过 volume 将宿主机目录挂载到该路径，后端才能访问

### 2.5 涉及文件

- **页面**：`frontend_spa/src/pages/FolderSync.tsx`
- **类型与 API**：见上文
- **类型声明**：`frontend_spa/src/vite-env.d.ts` 中为 File System Access API 增加 `Window.showDirectoryPicker` 及 `FileSystemDirectoryHandle` 的声明

## 三、Docker 部署友好化方案（方案 B）

### 3.1 问题

用户在前端选择的是**本机（或宿主机）上的文件夹路径**，而后端运行在 **Docker 容器内**，只有容器内路径对后端可见。若不做挂载，用户选的本地路径在容器中无法访问。

### 3.2 方案对比

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **A** | 用户自己配置「本地路径 → 容器内路径」映射 | 灵活 | 对普通用户有难度，需理解挂载概念 |
| **B** | 系统自动生成「容器内路径」并给出 Docker volume 配置命令，用户复制执行即可 | 操作简单，适合普通用户 | 依赖用户按文档执行一次挂载与重启 |

### 3.3 推荐方案 B 的设计要点（已实现）

1. **用户选择本地文件夹**  
   前端通过「选择文件夹」得到本地路径（或文件夹名称，视浏览器能力而定）。

2. **自动分配容器内路径**  
   点击「生成容器路径」按钮，系统自动生成唯一容器内路径：  
   `/data/sync/{知识库ID}`，避免多知识库冲突。

3. **生成 Docker 配置命令**  
   页面自动展示 Docker 挂载命令，包含：
   - `docker run -v` 命令示例
   - `docker-compose.yml` volumes 片段
   - 提供「一键复制」按钮

4. **用户操作步骤**  
   - 在前端选择知识库，点击「生成容器路径」自动填充容器内路径；
   - 页面展示 Docker 命令，点击复制按钮；
   - 在宿主机执行命令，或更新 compose 并重启；
   - 重启后端/worker 容器使挂载生效；
   - 返回页面点击「立即同步」测试。

5. **涉及文件**
   - **工具函数**：`frontend_spa/src/lib/docker.ts`（路径生成、命令生成、复制功能）
   - **页面**：`frontend_spa/src/pages/FolderSync.tsx`（UI 交互与展示）

### 3.4 界面说明

在「文件夹同步」页面的配置表单中：
- 同步目录输入框右侧新增「生成容器路径」按钮
- 点击后自动填充 `/data/sync/{kbId}` 格式的路径
- 同时展示 Docker 挂载命令卡片（黄色背景提示框）
- 卡片内包含 docker run 命令和 docker-compose 片段
- 每个命令旁都有「复制」按钮
