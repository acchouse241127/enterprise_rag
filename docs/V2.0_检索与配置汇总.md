## 二、环境配置（.env）与 LLM 任务映射

### 2.1 .env 版本与区块

- **版本**：`APP_VERSION=2.0.0`，标题注明「检索与质量升级」。
- **主要区块**：应用基础、数据库、Redis/Celery、文档处理、V2.0 检索（含 retrieval_mode、RRF_K、自适应深度、去噪）、Reranker、去重与动态阈值、LLM、V2.0 质量保障、检索日志、文件夹同步、安全、Docker、备选 LLM 示例。

### 2.2 LLM 任务类型与前端功能对应

| task_type | 前端位置 | 功能说明 |
|-----------|----------|----------|
| `qa` | 问答页 → 提问框 | 主问答：用户提问后，由 LLM 生成答案 |
| `query_expansion` | 问答页 → ⚙️ 检索设置 → **查询扩展模式** | 查询扩展：对用户问题做规则/LLM/混合扩展后再检索 |

### 2.3 LLM_TASK_OVERRIDES 三场景

| 场景 | 配置含义 | 前端行为 |
|------|----------|----------|
| 场景 1 | 为 `query_expansion` 指定与主 LLM 不同的云端模型（如另一套 Kimi 配置） | 查询扩展用指定模型，问答用默认 LLM |
| 场景 2 | 为 `query_expansion` 指定本地 Ollama/vLLM | 查询扩展用本地（省 API），问答用默认 LLM |
| 场景 3 | 不配置 `LLM_TASK_OVERRIDES` | 查询扩展与问答都使用默认 LLM |

## 三、检索五要素：关系与联动（混合方案）

> **V2.0 采用混合方案**：普通用户只选策略（内置最优配置），高级用户展开高级选项可覆盖。

### 3.1 策略定义（V2.0 混合方案）

| 策略名 | 显示名 | TopK | 扩展模式 | 检索模式 | 适用场景 |
|--------|--------|------|----------|----------|----------|
| `smart` | 智能（推荐） | 8 | hybrid | hybrid | 平衡召回与精度，适合大多数场景 |
| `precise` | 精准 | 5 | none | hybrid | 优先答案准确性，减少噪音 |
| `fast` | 快速 | 4 | none | vector | 优先响应速度，适合实时场景 |
| `deep` | 深度 | 12 | llm | hybrid | 最大召回范围，适合探索性查询 |

### 3.2 五要素定义

1. **检索策略**：smart / precise / fast / deep（预设所有参数组合）。
2. **检索模式**：vector / bm25 / hybrid（V2.0，控制 BM25+向量+RRF）。
3. **检索数量（TopK）**：最终返回的 chunk 数量，可覆盖策略默认值。
4. **回答风格**：system_prompt_version（A/B/C），仅影响 Prompt 模板。
5. **查询扩展模式**：rule / llm / hybrid，决定如何扩展用户问题。

### 3.3 UI 设计：渐进式披露

```
┌─────────────────────────────────────────────────────┐
│  检索策略                                            │
│  ○ 智能（推荐）  ○ 精准  ○ 快速  ○ 深度              │
│                                                      │
│  回答风格: [C 折中 ▼]                               │
│                                                      │
│  ▼ 高级选项（自定义检索模式、扩展方式、数量）         │
└─────────────────────────────────────────────────────┘
                    ↓ 点击展开
┌─────────────────────────────────────────────────────┐
│  检索模式: [混合检索 ▼]  (策略默认：混合检索)         │
│  查询扩展: [混合版 ▼]    (策略默认：混合版)           │
│  检索数量: [8 ▼]         (策略默认：8)               │
└─────────────────────────────────────────────────────┘
```

### 3.4 联动逻辑

| 配置对 | 联动行为 |
|--------|----------|
| 策略 → 查询扩展模式 | 若策略 `expansion_enabled=false`，禁用扩展选择器并提示 |
| 策略 → 高级选项默认值 | 高级选项显示「策略默认：xxx」提示 |
| 高级选项 → 策略 | 用户覆盖后不影响策略选择，保持独立性 |

### 3.5 绝对联动结论

| 配置对 | 是否绝对联动 | 说明 |
|--------|--------------|------|
| 策略 → 查询扩展模式 | **是** | 当策略为 high_precision 或 low_latency 时，`expansion_enabled=false`，扩展模式不生效，前端应禁用扩展选择器并提示 |
| 策略 → 检索数量 | 否 | 策略只提供默认 top_k，用户可覆盖 |
| 策略 → 检索模式 | 否 | 独立 |
| 策略 → 回答风格 | 否 | 独立 |
| 检索数量 / 回答风格 → 其他 | 否 | 独立 |

## 四、联动逻辑实现：后端 / API / 前端

### 4.1 后端（已完成 ✅）

- **文件**：`backend/app/rag/retrieval_strategy.py`
- **改动**：
  - `RetrievalStrategy` 新增 `display_name`、`expansion_mode`、`retrieval_mode` 字段
  - 策略名改为 `smart` / `precise` / `fast` / `deep`（面向用户更友好）
  - `list_strategies()` 返回完整配置

### 4.2 API 接口（无需改动）

- `GET /api/qa/strategies`：返回策略列表（含新增字段）
- `POST /api/qa/stream`：请求体保持不变

### 4.3 前端类型（已完成 ✅）

- **文件**：`frontend_spa/src/api/types.ts`
- **改动**：`Strategy` 接口增加 `display_name`、`expansion_mode`、`retrieval_mode`

### 4.4 前端组件联动（已完成 ✅）

- **文件**：`frontend_spa/src/components/qa/QASettingsSheet.tsx`
- **实现要点**：
  1. **高级选项开关**：默认折叠，点击展开
  2. **策略联动**：切换策略时，显示「策略默认：xxx」提示
  3. **扩展禁用**：`expansion_enabled=false` 时禁用扩展选择器并显示提示「当前策略已禁用扩展」
  4. **本地兜底**：`STRATEGIES_FALLBACK` 保证离线时联动仍生效
  5. **useEffect 联动**：策略切换时自动重置无效的扩展模式

**关键代码片段**：

```tsx
// 策略是否启用扩展
const expansionEnabled = currentStrategy?.expansion_enabled ?? true;

// 当策略禁用扩展时，禁用选择器并提示
<Select
  value={settings.queryExpansionMode}
  onValueChange={(v) => update("queryExpansionMode", v as QASettings["queryExpansionMode"])}
  disabled={!expansionEnabled}  // 关键：禁用状态
>
  ...
</Select>
{!expansionEnabled && (
  <p className="text-xs text-muted-foreground">
    「{currentStrategy?.display_name}」策略已禁用查询扩展，选择器不可用
  </p>
)}
```

## 五、Kimi Provider 与 .env 使用要点

- **Provider**：`LLM_PROVIDER=kimi`，`LLM_MODEL_NAME=moonshot-v1-8k`，`LLM_BASE_URL=https://api.moonshot.cn/v1`，API Key 填 `LLM_API_KEY`。
- **Temperature**：Kimi 仅支持 1.0，代码中 `KimiProvider` 已写死，.env 中 `LLM_TEMPERATURE=1.0` 仅作说明。
- **可选**：通过 `LLM_TASK_OVERRIDES` 为 `query_expansion` 指定不同模型（如本地 Ollama），实现问答用 Kimi、扩展用本地的组合。
