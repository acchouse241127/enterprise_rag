# 保姆式使用说明：零基础用 ECC + Superpowers 优化项目

**适合谁**：完全没有编程基础，只想按步骤操作就能让项目变更好。  
**你需要做的**：用 Cursor 打开项目，在对话框里**复制粘贴下面给出的句子**，发送给 AI，等它做完再执行下一步。  
**你不用做的**：不用自己写代码、不用敲命令、不用懂技术名词（文中会简单解释）。

---

## 一、开始前准备（只做一次）

### 1.1 确认两件事

1. **项目已经按「方案 B」装好了 ECC 和 Superpowers**  
   如果还没装，在 Cursor 里对 AI 说一句：
   > 请按 docs/Cursor-ECC-Superpowers-安装方案.md 里的方案 B，帮我执行一键安装脚本。

2. **用 Cursor 打开的是「项目文件夹」**  
   - 菜单选 **文件 → 打开文件夹**（或 File → Open Folder）。  
   - 选 **enterprise_rag** 这个文件夹（不要选上一级的 Super Fund）。  
   - 左侧文件树里能看到 `backend`、`frontend`、`docs`、`.cursor` 等，就说明打开对了。

### 1.2 每次优化时怎么操作

- 在 Cursor 里点 **聊天 / Chat**（或按快捷键打开对话）。  
- **每次新开一个对话**做「一整轮」优化（例如：这一轮只做「安全审查」）。  
- 把下面**步骤里给你的「你要发的那句话」**整段复制，粘贴到输入框，点发送。  
- 等 AI 回复并做完它说的事（有时会改文件、有时只给你报告）。  
- 做完一步，再复制下一步的句子，继续。

**重要**：AI 可能会问你「是否允许修改文件」。你点**允许**即可，它才会帮你改代码或文档。

---

## 二、优化总览（你要做的几件「大事」）

按顺序做下来，就是从「先看看哪里有问题」到「整体优化一轮」的完整流程：

| 顺序 | 做什么 | 目的（用人话说） |
|------|--------|------------------|
| 第 1 步 | 全面评估 + 安全审查 | 让 AI 先帮你查一遍：项目哪里可能不安全、哪里可以更好，并出一份报告或顺带修掉明显问题。 |
| 第 2 步 | 代码规范与 Python 审查 | 让 AI 按 ECC 的规则检查后端代码风格、类型、错误处理，并给出修改或直接小改。 |
| 第 3 步 | 测试与覆盖率 | 让 AI 看现有测试缺什么、覆盖率多少，并帮你补测试，目标更接近「改代码不容易踩坏老功能」。 |
| 第 4 步 | 文档更新 | 让 AI 根据当前项目状态更新 README、用户手册等，方便以后自己或别人看懂。 |
| 第 5 步 | 发布前验证 | 让 AI 按「发布前检查清单」跑一遍（测试、安全、规范），确保可以安心发布或合并。

下面每一节里都有**你要发的那句话**，直接复制粘贴即可。

---

## 三、第 1 步：全面评估 + 安全审查

**目的**：让 AI 用 ECC 的「安全审查」和「项目评估」能力，先给项目做一次全面体检，并重点查安全问题（例如：有没有把密钥写在代码里、接口有没有限流等）。

### 你要发的第一句话（复制下面整段）

```
请严格按照 .cursor 里的 ECC 和 Superpowers 规则来执行。

请用 ECC 的 security-reviewer 代理（或 security-review 技能），对 enterprise_rag 项目做一次全面的安全审查，重点包括：
1. 是否有硬编码的密钥、API Key、密码；
2. 用户输入是否都有校验（上传、登录、问答接口）；
3. 认证与限流是否到位；
4. 错误信息是否可能泄露敏感信息。

请先扫描 backend 和 .env.example，再给出一份「安全审查报告」（用中文），并列出建议修改项。如果是明显问题，请直接帮我改代码；如果不确定，只列出建议让我确认。
```

**AI 会做什么**：根据 `.cursor` 里的安全规则和 security-reviewer/security-review 来检查代码和配置，给你一份中文报告，并可能直接改掉明显问题（例如注释里误写的密钥）。

**怎样算完成**：你收到一份带条目的「安全审查报告」，并且 AI 要么已经改了它说的明显问题，要么列出了「建议修改」清单。若 AI 问「是否允许修改文件」，选**允许**。

---

### 你要发的第二句话（可选：想要更完整的「体检」时用）

如果除了安全，你还想顺便看「整体架构、测试、文档哪里可以加强」，可以在安全审查做完后，再发下面这句：

```
请再根据 docs/ECC与Superpowers工具评估报告.md 里对 enterprise_rag 的优化映射，做一次「整体优化评估」：
1. 代码规范与风格（是否符合同一风格、单文件是否过长）；
2. 测试与覆盖率（缺哪些测试、覆盖率大概多少）；
3. 数据库与 RAG（Postgres/ChromaDB 使用是否合理）；
4. 文档（README、用户手册是否和当前功能一致）。

用中文给出一份「整体优化评估报告」，每条都注明：现状、建议、是否建议在本轮就改（是/否）。只建议「是」的再动手改，其它只列出建议。
```

**怎样算完成**：你收到一份「整体优化评估报告」，并且对建议「是」的项，AI 已经改过代码或文档。

---

## 四、第 2 步：代码规范与 Python 审查

**目的**：让 AI 按 ECC 的 Python 规则（编码风格、类型、错误处理等）审查后端代码，并自动做能达成共识的修改（例如格式、简单类型注解）。

### 你要发的话（复制下面整段）

```
请严格按照 .cursor 里的 ECC 规则执行，用 python-reviewer 代理（或按 python-review 命令）对 enterprise_rag 的 backend 做一次「Python 代码审查」：

1. 遵循 .cursor/rules 里的 python-coding-style、python-patterns、common-coding-style；
2. 检查：PEP 8 风格、类型注解、错误处理、输入校验、单文件是否过长（例如超过 800 行）、是否有不该有的可变修改；
3. 给出简短的中文审查报告（每条问题注明文件和行号或模块名）；
4. 对明确的风格/类型/简单错误处理问题直接改；对需要业务判断的只列建议，等我确认再改。
```

**AI 会做什么**：按 ECC 的 Python 规则扫描 `backend`，给你一份审查报告，并直接改掉「明确可改」的部分。

**怎样算完成**：你收到审查报告，且 AI 对「明确可改」的已经改完。若它列出「建议你确认」的项，你可以回复「这些也按你的建议改」或「先不改」。

## 五、第 3 步：测试与覆盖率

**目的**：让 AI 看现有测试缺什么、覆盖率多少，并按 ECC 的「测试要求」和「TDD/覆盖率」能力补测试，让项目更稳。

### 你要发的话（复制下面整段）

```
请按 .cursor 里的 ECC 测试规则和 python-testing、tdd-workflow 技能来操作：

1. 先分析 enterprise_rag backend 的测试现状：当前有哪些测试、缺哪些模块或接口的测试、覆盖率大概多少（如能跑 pytest --cov 请跑一下）；
2. 给出一份中文的「测试与覆盖率报告」：已覆盖/未覆盖的模块、建议优先补哪些测试；
3. 按优先级帮我补写测试（优先：核心 API、认证、RAG 相关），遵循项目里已有的 conftest 和 pytest markers（integration/slow/llm），目标是把覆盖率提高到 80% 左右或至少明显提升。
```

**AI 会做什么**：分析 `backend/tests` 和代码结构，跑或说明如何跑覆盖率，给你报告，然后按优先级补写测试文件。

**怎样算完成**：你收到「测试与覆盖率报告」，并且多了或改动了若干测试文件。若 AI 说「需要本地跑 pytest」，而你本机没环境，可以回一句「我本地暂时不跑测试，你只要把测试代码写好并说明如何在 CI 里跑即可」。

## 六、第 4 步：文档更新

**目的**：让 AI 根据当前代码和功能，更新 README、用户手册等，方便以后自己或别人看懂项目。

### 你要发的话（复制下面整段）

```
请按 ECC 的 doc-updater 代理（或 update-docs 命令），根据当前 enterprise_rag 的代码和功能，更新以下文档：

1. 项目根目录的 README.md：确保「功能特性、技术栈、快速启动、项目结构、配置说明、常见问题」和当前一致；
2. docs/用户使用手册.md（如存在）：确保和实际界面、操作一致；
3. 若有明显过时的 API 或环境变量说明，请一并改掉。

用中文在回复里简要列出「已更新哪些文件、主要改了哪些内容」。
```

**AI 会做什么**：对照当前项目结构和配置，更新 README 和用户手册，并给你一个更新摘要。

**怎样算完成**：你收到更新摘要，并且 README、用户手册等已是最新内容。

## 七、第 5 步：发布前验证

**目的**：在你要「发布」或「合并到主分支」之前，让 AI 按 ECC + Superpowers 的「发布前检查」跑一遍，确保测试、安全、规范都过关。

### 你要发的话（复制下面整段）

```
请按 .cursor 里 ECC 的 verify、test-coverage 以及 Superpowers 的 verification-before-completion 来做一次「发布前验证」：

1. 检查清单：测试是否能通过、是否有硬编码密钥、关键接口是否有测试、文档是否已更新；
2. 若项目里能跑 pytest，请说明如何运行单元测试和集成测试，以及如何查看覆盖率；若不能在本环境跑，请列出「发布前建议在本地或 CI 执行的命令」；
3. 给出一份中文的「发布前验证报告」：通过项 / 未通过项 / 建议项，并说明未通过或建议项要怎么处理。
```

**AI 会做什么**：按验证清单检查项目，给你一份「发布前验证报告」，并说明需要执行的命令或需要你手动确认的项。

**怎样算完成**：你收到「发布前验证报告」。若有「未通过」或「建议」项，按 AI 说明在本地或 CI 里执行（或再发一句让 AI 帮你改）。

## 八、一次做完「一整轮优化」（可选）

如果你希望**一次对话里按顺序做完 1～5 步**，可以直接发下面这句（适合时间充裕、希望一气呵成时用）：

```
请严格按照 .cursor 里的 ECC 和 Superpowers 规则，对 enterprise_rag 做一整轮优化，按顺序执行并每步给出简短中文小结：

1. 安全审查：用 security-reviewer/security-review 做全面安全审查，出报告并修掉明显问题；
2. 代码审查：用 python-reviewer 按 ECC Python 规则审查 backend，出报告并做明确可改的修改；
3. 测试与覆盖率：分析测试现状和覆盖率，出报告并按优先级补测试；
4. 文档更新：用 doc-updater/update-docs 更新 README 和用户手册；
5. 发布前验证：按 verification-before-completion 和 verify 给出一份发布前验证报告。

每步完成后用「第 X 步完成：…」的格式小结，最后给一份「本轮优化总结」。
```

**注意**：一整轮可能会改很多文件，对话也会很长。若你更想「一步一步确认」，建议还是按上面第三～七节，分 5 次发、分 5 步做。

## 九、遇到这些情况怎么办

| 情况 | 你可以怎么做 |
|------|--------------|
| AI 没有按 ECC/Superpowers 做 | 在下一句里强调：「请严格按照 .cursor 里的 ECC 和 Superpowers 规则和技能执行，先读相关 rules 和 skills 再动手。」 |
| AI 问「是否允许修改文件」 | 点**允许**，它才能帮你改代码或文档。 |
| 改完之后发现不满意 | 可以说：「把刚才改的 XXX 文件恢复成修改前的样子」或「只保留 XXX 修改，其它还原」。若项目在 Git 下，也可以让 AI 教你用「撤销」或「还原文件」。 |
| 不知道某一步算不算做完 | 看每步的「怎样算完成」：收到报告 + AI 已做完它说会做的修改，即算完成。 |
| 我完全不想改代码，只要报告 | 在第 1、2、3、5 步的提示里加上：「只出报告，不要直接改代码。」这样 AI 只会给你评估/审查报告，不会动文件。 |

## 十、一句话速记（方便以后用）

- **只想查安全**：发「用 ECC 的 security-reviewer 对 enterprise_rag 做一次全面安全审查，出中文报告并修掉明显问题。」
- **只想查代码规范**：发「用 ECC 的 python-reviewer 按 .cursor 规则审查 backend，出报告并改掉明确问题。」
- **只想补测试**：发「按 ECC 的 python-testing 和 tdd-workflow 分析 backend 测试和覆盖率，出报告并优先补核心 API 和认证的测试。」
- **只想更新文档**：发「按 ECC 的 doc-updater 根据当前项目更新 README 和用户手册，并列出更新摘要。」
- **发布前检查**：发「按 verification-before-completion 和 verify 做发布前验证，出中文报告并列出要执行的命令或建议。」
