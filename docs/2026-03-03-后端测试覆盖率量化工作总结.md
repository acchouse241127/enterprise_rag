## 技术债务与挑战

### 1. NumPy导入冲突 ⚠️
**问题**:
```
ImportError: cannot load module more than once per process
```
**原因**: sentence_transformers、transformers、numpy之间的循环导入冲突
**影响**: 无法在同一进程运行包含这些依赖的测试

**解决方案**:
- 使用独立子进程运行测试
- 使用Mock避免实际导入
- 重构代码减少循环依赖

### 2. 测试环境配置 ⚠️
**问题**: 测试需要访问数据库、Redis、向量数据库等外部服务
**影响**: 部分测试需要完整环境

**解决方案**:
- 全面的Mock策略
- 测试专用配置文件
- CI/CD环境变量管理

### 3. API不匹配 ⚠️
**问题**: 编写的测试与实际代码API不匹配
**示例**: RetrievalOrchestrator构造函数参数不匹配
**影响**: 测试失败，需要调整

**解决方案**:
- 先读取实际代码再编写测试
- 使用类型检查工具验证
- 增量测试编写

## 测试编写最佳实践总结

### 1. TDD方法论
```
RED → GREEN → REFACTOR
```
- 先写失败的测试
- 实现最小代码使测试通过
- 重构保持测试通过

### 2. Mock策略
```python
# Mock外部依赖
@patch("app.cache.query_cache.redis.from_url")
def test_redis_connection(mock_from_url):
    mock_redis = Mock()
    mock_from_url.return_value = mock_redis
    # 测试逻辑
```

### 3. 测试组织
```python
class TestClassName:
    """类级文档字符串"""

    def test_feature_behavior(self):
        """测试功能行为的文档字符串"""
        # Arrange-Act-Assert模式
        # Arrange: 准备测试数据
        # Act: 执行被测功能
        # Assert: 验证结果
```

### 4. 边界测试
- 空输入（None, 空字符串, 空列表）
- 边界值（0, 1, 最大值）
- 异常情况（网络错误, 超时）
- 多个相同类型PII

### 5. Fixture使用
```python
@pytest.fixture
def mock_redis():
    """可复用的Mock对象"""
    mock = Mock()
    mock.get.return_value = "cached_data"
    return mock

def test_with_fixture(mock_redis):
    # 使用fixture
    result = mock_redis.get("key")
    assert result == "cached_data"
```
