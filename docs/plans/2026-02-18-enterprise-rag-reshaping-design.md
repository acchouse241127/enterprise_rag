## 一、需求摘要（澄清结果）

| 维度 | 结论 |
|------|------|
| 目标用户 | 先团队使用，成熟后对外（订阅/买断） |
| 部署 | 测试阶段 Docker（本机多人访问）；未来自建/上云按用量与成本权衡；ToB 订阅或定制均需支持灵活部署 |
| 登录 | 测试阶段账号+密码；后续可加双因素 |
| 文档与输入 | 现有格式 + 音频（常用）+ 视频（常用）+ 国内外网址解析；单文件与处理在资源约束下平衡，预留升级空间 |
| 迭代范围 | 本轮包含音频、视频、网址；不急于上线，有备份且已在 Docker 运行 |
| 问答体验 | 流式输出（尽量高效）+ LLM 不可用时降级展示（仅检索结果，响应尽量快） |
| 可发布标准 | 核心链路跑通 + 自动化测试/回归 + **团队手工验收**；持续提升解析→检索→回答满意度；检索质量看板与联动功能需在本轮或后续展开 |

**实施策略**：方案 B（并行推进）— 本轮同时推进新输入类型（音视频网址）、检索与问答质量（解析/检索精度与满意度、看板）与核心链路，按模块并行，最后统一验收。

## 二、架构与部署

- **选型**：**二 + 预留三** — API + 异步 Worker + 队列（Redis + Celery 或等价），解析/Worker 模块化，预留拆为独立服务边界。
- **部署**：Docker Compose：postgres、chromadb、redis、backend（API）、worker、frontend（见前端选型）；扩量时先加 Worker，再考虑自建/上云。

---

## 三、认证与用户

- V1/测试阶段：账号+密码，JWT 签发；用户表与现有一致；预留 TOTP 字段与接口；密码哈希、登录限流、JWT 从环境变量读取。

---

## 四、知识库、上传与输入

- 知识库/文档表与现有一致；上传与解析异步化，API 写 Document 状态、发任务入队，Worker 执行 parse→chunk→embed→upsert。
- **输入类型**：现有文档格式 + **音频**（转写文本）+ **视频**（抽帧/字幕/转写）+ **网址**（抓取正文去噪）；大文件与资源平衡，预留升级开关。
- **解析**：方案一（解析与分块增强）+ 方案二（检索链路增强）+ 方案三（端到端闭环）并行；解析逻辑模块化，预留方案三（独立 Parser 服务）。

---

## 五、检索与 RAG

- **检索与 RAG**：三版组合实施 — 版一（多路召回+查询扩展）→ 版二（多阶段排序+可配置策略）→ 版三（引用约束+可解释埋点）。
- 流式与降级：保持现有流式接口与降级逻辑；目标为流式效率高、降级响应快。
- 闭环与看板：RetrievalLog 与反馈扩展，支撑检索质量看板与满意度分析（方案三）。

---

## 六、前端

- **选型**：**方案二 — 独立 SPA**（React / Vue / Next.js）+ 现有 FastAPI；Streamlit 可保留内部/调试或逐步下线。
- 实施计划含：前端技术栈选定、工程初始化、与现有 API（鉴权、流式、降级）对接、登录/知识库/文档上传/问答/看板/异步任务等页面迁移或新建、部署与 CI 中前端构建与后端分离。

---

## 七、后端（相对现有 FastAPI）

- **选型**：**后端方案二** — SPA 对接与契约（CORS、鉴权约定、流式/错误格式）+ 队列+Worker、解析与分块增强、检索与 RAG 三版组合、闭环与看板。
- **后端方案三**（多租户、限流/配额、审计）：**本轮不实现**，在设计与代码上**预留空间**（如租户/组织扩展点、限流与审计接口或配置预留、文档注明 ToB 就绪阶段接入）。

---

## 八、验收、错误处理与测试

- **验收**：核心链路 + 自动化测试/回归 + 团队手工验收；看板可看、反馈可录。
- **错误处理**：API 统一异常、生产不泄露堆栈；解析/Worker 失败写回状态；前端降级与明确提示。
- **测试**：单元（覆盖率与 ECC 对齐）、集成（markers）、E2E 或人工关键路径；发布前全量回归。

## 九、档三（全面 LLM）目标与模型配置

**档三目标陈述**：采用档三（全面引入 LLM）时，目标为：**在解析、检索、重排等模块均已优化到极致的前提下，通过查询改写、可解释、摘要、多轮摘要、grounding 校验等环节，使送达「内容生成 LLM」的输入达到当前系统可提供的最优状态，从而在现有 RAG 架构下对答案生成质量尽最大努力；实际满意度仍受模型能力、知识库质量与成本约束。**

**档三下模型配置策略**（二选一或分阶段）：

| 策略 | 说明 | 适用 |
|------|------|------|
| **A. 单模型多角色** | 所有 LLM 调用同一套 API/同一模型，用不同 prompt 区分用途；配置简单、易维护。 | 起步与资源有限时优先。 |
| **B. 多模型分工** | 快/小模型：查询改写、摘要、可解释、反馈理解、错误建议、情境帮助、审核等；强/大模型：RAG 回答生成、grounding 校验、推荐追问。 | 延迟与成本优化、核心回答质量优先时按需引入。 |

**实施顺序**：**先按策略 A 实现档三**，根据测试结果（延迟、成本、满意度）再决定是否拆成策略 B。

**策略 B 的开发迭代空间（预留）**：

- **配置**：在配置/环境变量中预留「按任务类型指定 model / base_url / api_key」的扩展项；策略 A 下不填即全部使用默认同一模型。
- **代码**：所有 LLM 调用经统一入口（如 `get_provider_for_task(task_type)`）获取 provider；默认行为为「任意 task_type 均返回同一 provider」（策略 A）；策略 B 时仅需在配置中为部分 task 指定不同 model/endpoint，工厂按 task_type 返回对应 provider，无需改业务调用方。
- **任务类型枚举**：预留与档三 13 处调用点对应的 task_type（如 `qa`、`rewrite`、`summary`、`explain`、`grounding`、`session_summary`、`feedback`、`error_suggestion`、`suggest_question`、`follow_up`、`context_help`、`moderation` 等），便于策略 B 时按类型路由到快/大模型。

档三下共 13 处 LLM 调用点（见过程记录「按档三方案时，系统模型配置概览」表）。
