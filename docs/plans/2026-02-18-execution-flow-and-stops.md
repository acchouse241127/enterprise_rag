## 三、运行流程（从开始到结束）

1. **开始前**
   - 读取实施计划，做一次**关键性审阅**（若有明显缺口或风险会先向你提出）。
   - 若使用 worktree：按 using-git-worktrees 创建隔离工作区与功能分支（如 `reshaping-v1`），后续所有改动在该分支上进行。

2. **执行批次（循环）**
   - 每批按计划执行 2～3 个 Task（与上表批次对应）。
   - 每个 Task 内：按计划步骤执行；能写测试的先写再实现；按计划做验证（如运行 pytest、启动服务等）。
   - 本批所有任务验证通过后，进行**报告**（实现了什么、验证命令与结果）。

3. **停靠点（两种模式，请你二选一）**

   - **模式 A — 每批停一次（默认，符合 executing-plans 原意）**  
     - 每完成**一个批次**就**停下来**，向你报告并说「Ready for feedback」。  
     - 你回复「继续」或给出修改意见后，再执行下一批。  
     - **停靠次数**：共 **7 次**（批次 1～7 各一次），最后再在「finishing」处停一次。

   - **模式 B — 全自动跑到完再停**  
     - **不**在每批之间等待反馈，连续执行批次 1～7 直至全部 24 个 Task 完成。  
     - 仅在**全部任务完成并验证通过**后，进入下一步「完成开发」，并在那里停一次。

4. **完成开发（finishing-a-development-branch）**
   - 全量运行测试（如 `pytest`），确认通过。
   - 使用 **finishing-a-development-branch**：呈现 **4 个选项**：
     1. 合并回 main（本地 merge）  
     2. 推送并创建 Pull Request  
     3. 保留当前分支与工作区，稍后自行处理  
     4. 丢弃本分支及工作区  
   - **在这里再次停下**，等待你选择 1 / 2 / 3 / 4（若选 4 需输入 `discard` 确认）。  
   - 根据你的选择执行合并 / 创建 PR / 保留 / 丢弃，并按要求清理或保留 worktree。

5. **异常停靠**
   - 若某一批执行中遇到**阻塞**（依赖缺失、测试持续失败、指令不清等），会**立即停止**并说明问题，向你请求帮助，不会自行猜测通过。
