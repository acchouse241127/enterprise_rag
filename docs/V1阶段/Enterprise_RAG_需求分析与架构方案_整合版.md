# Enterprise RAG 需求分析与架构方案（整合版）

> 本文档整合「版本 B 深化方案」与「一步到位补充优化」，涵盖需求分析、架构方案、实施计划与补充优化。面向团队自用，暂不考虑 ToB。

**编制日期**：2026-02-21  
**编制角色**：RAG 系统软件架构师

## 第一部分：需求分析

### 1.1 目标用户与使用场景

| 维度 | 结论 |
|------|------|
| **目标用户** | 团队内部使用，成熟后可考虑对外 |
| **部署方式** | 测试阶段 Docker（本机多人访问）；未来可自建/上云 |
| **登录** | 账号+密码，JWT 签发；TOTP 双因素已预留 |
| **ToB** | 本轮不实现，设计上预留 tenant_id 等扩展点 |

### 1.2 功能需求

| 需求 | 说明 |
|------|------|
| **文档输入类型** | 现有文档格式 + 音频（mp3、wav 等）+ 视频（mp4、webm 等）+ 国内外网址解析；单文件上限 200MB |
| **问答体验** | 流式输出（打字机效果）+ LLM 不可用时降级展示（仅检索结果，响应尽量快） |
| **引用溯源** | 回答自动标注来源 `[ID:x]`，支持 reason、snippet 可解释 |
| **可发布标准** | 核心链路跑通 + 自动化测试/回归 + 团队手工验收；检索质量看板与联动功能 |
| **迭代范围** | 本轮包含音频、视频、网址；解析→检索→回答满意度持续提升 |

### 1.3 非功能需求

| 需求 | 说明 |
|------|------|
| **部署** | Docker Compose 编排；扩量时先加 Worker，再考虑自建/上云 |
| **安全** | 密码 bcrypt、JWT、TOTP、生产异常不泄密、限流 |
| **可维护性** | 模块化、清晰接口、文档完善、易于测试 |
| **可观测** | Prometheus 指标、操作日志、健康检查 |

### 1.4 实施策略

**方案 B（并行推进）**：本轮同时推进新输入类型（音视频网址）、检索与问答质量（解析/检索精度与满意度、看板）与核心链路，按模块并行，最后统一验收。

## 第二部分：架构方案

### 2.1 系统整体架构

```
用户 → Nginx（生产）→ FastAPI API → Redis 队列 → Celery Worker
  |           |              |                          |
  |           |              └── PostgreSQL + ChromaDB + 本地文件
  |           |
  └── SPA 前端（主）/ Streamlit（内部调试）
```

### 2.2 技术选型

| 层级 | 技术 | 说明 |
|------|------|------|
| 后端 | FastAPI | 高性能异步 API |
| 队列 | Redis + Celery | 任务持久化、失败重试、Worker 水平扩展 |
| 关系库 | PostgreSQL | 用户、知识库、文档、检索日志等 |
| 向量库 | ChromaDB | 向量索引、文档 Embedding |
| Embedding | BGE-M3 | 本地/远程，中英文双语 |
| LLM | DeepSeek / OpenAI / Gemini 等 | 兼容接口 |
| 前端 | React + Vite + TypeScript（SPA） | 主入口；Streamlit 保留内部/调试 |

### 2.3 核心模块职责

| 模块 | 职责 |
|------|------|
| **API** | 认证、知识库、文档、问答、检索日志、反馈、任务；CORS、鉴权、流式/错误契约 |
| **Worker** | 消费解析任务：parse → chunk → embed → upsert；失败写回 parse_failed |
| **解析** | 文档、音频、视频、网址解析器；模块化，可预留独立 Parser 服务 |
| **检索** | 多路召回（向量+关键词）、查询扩展、去重、Reranker、动态阈值、可配置策略 |
| **前端 SPA** | 登录、知识库、文档上传、URL 导入、问答、看板、任务页；流式、引用、降级 |

### 2.4 文档处理流程

```
用户上传 → API 写 Document(status=pending) → parse_and_index.delay(id) → 返回 201
                    ↓
            Worker 消费 → parse → chunk → embed → upsert → 更新 status=vectorized
                    ↓
            前端轮询 GET /documents/{id} 查 status
```

### 2.5 RAG 问答流程

```
用户提问 → 多路召回（向量+关键词）→ 查询扩展（可选）→ 去重 → Reranker → 动态阈值
    → 构建 Prompt → LLM 流式生成 → insert_citations → 返回回答 + citations
    → 若 LLM 失败：降级返回 retrieved_chunks
```

## 第三部分：版本 B 深化方案

### 3.1 批次概览

| 批次 | 内容 | 预估 |
|------|------|------|
| 批次 1 | 后端 SPA 对接（CORS、鉴权、流式/错误契约） | 已大部分完成 |
| 批次 2 | 队列与 Worker 基础（Redis、Celery、异步上传） | 已完成 |
| 批次 3 | 解析增强（音视频、网址、分块策略） | 已部分完成 |
| 批次 4 | 检索与 RAG 三版（多路召回、查询扩展、策略、引用可解释） | 需深化 |
| 批次 5 | 闭环与看板（RetrievalLog、反馈、看板聚合） | 已基本完成 |
| 批次 6 | 前端 SPA 功能补齐 | 需深化 |
| 批次 7 | 验收与发布 | 已有清单 |

### 3.2 深化任务明细

#### 批次 4：检索能力补齐

| 子任务 | 内容 | 预估 |
|--------|------|------|
| B4.1 | 启用多路召回：retrieval_use_keyword 默认 True，验证效果 | 1h |
| B4.2 | 实现 Query Expansion：规则版（同义词/停用词）或 LLM 版，配置开关 | 2~3h |
| B4.3 | 可配置检索策略：high_recall / high_precision / low_latency，对应 top_k、expansion、keyword | 4h |
| B4.4 | API 支持 strategy 参数：POST /api/qa 可选传入 | 0.5h |

**策略预设**：

| 策略 | top_k | expansion | keyword | reranker_candidate_k |
|------|-------|-----------|---------|----------------------|
| default | 5 | 按配置 | 按配置 | 12 |
| high_recall | 8 | on | on | 16 |
| high_precision | 4 | off | off | 8 |
| low_latency | 4 | off | off | 6 |

#### 批次 4 补充：引用可解释

| 子任务 | 内容 | 预估 |
|--------|------|------|
| B4.5 | citations 增加 snippet：从 chunk 提取与 query 最相关子串 | 2h |
| B4.6 | RetrievalLog 记录 cited_chunk_ids：解析 answer 中的 [ID:x] | 2h |

#### 批次 3 补充：分块策略增强

| 子任务 | 内容 | 预估 |
|--------|------|------|
| B3.1 | 按 file_type 选 chunk mode：pdf/docx/txt/md → sentence；xlsx/pptx → char | 1.5h |
| B3.2 | parse_and_index_sync 中读取 KB 级 chunk_size/chunk_overlap | 1.5h |

#### 批次 6：前端 SPA 功能补齐

| 子任务 | 内容 | 预估 |
|--------|------|------|
| B6.1 | 知识库：创建、详情、文档列表 | 3h |
| B6.2 | 文档上传 + URL 导入，轮询 status | 3h |
| B6.3 | 问答页：SSE 解析 citations、llm_failed 降级展示 | 2.5h |
| B6.4 | 异步任务页、看板页图表 | 5h |
| B6.5 | SPA 纳入 Docker，作为主前端 | 2h |

### 3.3 执行路线建议

- **路线 A（检索优先）**：B4.1～B4.6 + B3.1～B3.2，约 15h
- **路线 B（全能力）**：批次 4 + 批次 3 补充 + 批次 6，约 31h+
- **路线 C（分阶段）**：先 B4+B3，验收后再 B6

## 第四部分：一步到位补充优化（非 ToB）

### 4.1 第一类：稳定与安全

| 项目 | 通俗解释 | 实现方式 | 预估 |
|------|----------|----------|------|
| 数据备份与恢复 | 像给电脑做系统备份，出问题可还原 | scripts/backup.py（PostgreSQL dump + data 打包）、restore.py；定时任务 | 4h |
| 生产网关 Nginx | 系统前加「门卫」，负责 HTTPS、分流、限流 | docker-compose 增加 nginx 服务；nginx.conf 反向代理 | 4h |
| 健康检查统一 | 给系统做「体检」，统一检查方式 | 文档说明：/api/system/health 探针、/health/ready 深度检查 | 1h |

### 4.2 第二类：敢改敢上线

| 项目 | 通俗解释 | 实现方式 | 预估 |
|------|----------|----------|------|
| 端到端自动化测试 | 脚本模拟真人操作，自动检查流程 | Playwright 覆盖登录→上传→问答；CI 中增加 E2E 任务 | 8h |
| 架构决策记录 ADR | 记录「为什么这样设计」 | docs/adr/ 目录；补写向量库、队列、前端选型等 ADR | 3h |
| 发布检查清单 | 上线前按清单逐项检查 | 完善 pre-release.md、manual-acceptance-v2.md | 1h |
| 故障排查手册 | 常见问题与排查步骤 | docs/故障排查手册.md | 1h |

### 4.3 第三类：好用与抗压

| 项目 | 通俗解释 | 实现方式 | 预估 |
|------|----------|----------|------|
| 大文件不拖慢小文件 | 大文件和小文件分开排队，小文件优先 | Celery task_routes，高/低优先级队列 | 4h |
| API 版本化 | 给接口加版本号，升级时旧客户端仍可用 | 路由前缀 /api/v1/，兼容期保留旧路径 | 2h |
| 向量库抽象 | 「存向量」做成可插拔模块 | VectorStore 接口，ChromaVectorStore 实现；预留 PgVectorStore | 6h |

### 4.4 第四类：文档与可维护

| 项目 | 通俗解释 | 实现方式 | 预估 |
|------|----------|----------|------|
| 架构文档更新 | 文档与系统实际一致 | 基于架构对照文档更新 Enterprise_RAG 架构方案 | 4h |
| 新人上手指南 | 如何跑起来、改哪里、怎么测 | docs/新人上手指南.md | 2h |
| 常见问题 FAQ | 常见问题与解答 | docs/FAQ.md | 2h |

### 4.5 第五类：体验

| 项目 | 通俗解释 | 实现方式 | 预估 |
|------|----------|----------|------|
| 上传进度显示 | 大文件上传时显示进度条 | 前端 progress 事件或分块上传 | 2h |
| 解析失败友好提示 | 解析失败时给出原因和建议 | Document.parser_message 细化；前端展示操作建议 | 2h |
| 资源可配置 | 可关闭耗资源解析（如视频），控制成本 | 配置开关：ENABLE_VIDEO_PARSE、ENABLE_AUDIO_PARSE 等 | 2h |

### 4.6 实施优先级（不含 ToB）

**第一批（约 10h）**：备份恢复、健康检查统一、ADR、发布清单、故障手册

**第二批（约 18h）**：Nginx、E2E、API 版本化、架构文档更新

**第三批（约 14h）**：任务分队列、向量库抽象、上手指南、FAQ

**第四批（按需）**：上传进度、解析失败提示、资源可配置

## 第五部分：实施路线与验收

### 5.1 执行顺序建议

1. **版本 B 深化**：优先完成批次 4（检索）+ 批次 3 补充（分块），再推进批次 6（SPA）
2. **一步到位第一批**：与版本 B 并行或穿插，完成备份、健康检查、ADR、发布清单、故障手册
3. **一步到位第二批**：版本 B 基本完成后，做 Nginx、E2E、API 版本化、架构文档
4. **第三、四批**：按资源与需求安排

### 5.2 验收标准

| 维度 | 标准 |
|------|------|
| 版本 B 批次 4 | 配置 strategy 后检索行为符合预期；citations 含 snippet |
| 版本 B 批次 6 | SPA 可完成登录→知识库→上传→问答→看板；流式与降级正常 |
| 一步到位第一批 | 备份脚本可执行；ADR 至少 3 篇；故障手册可跟随操作 |
| 一步到位第二批 | Nginx 可反向代理；E2E 覆盖关键流程；架构文档与实现一致 |
