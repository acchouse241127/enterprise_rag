## 二、从文档还原的「实际开发过程」时间线与阶段

### 2.1 版本与阶段（来自 README 版本历史）

| 版本 | 日期 | 说明 |
|------|------|------|
| v1.0.0 | 2026-02-13 | MVP 发布 |
| v2.0.0 | 2026-02-13 | Phase 2 完成：检索去重、动态阈值、RBAC、监控 |
| v3.1.0 | 2026-02-13 | Phase 3.1：文档版本管理、测试环境标准化 |
| v4.0.0 | 2026-02-14 | Phase 3 完成：异步任务、Prometheus/Grafana、对话导出分享、知识库在线编辑；单文件 200MB |

可推断：**V1（MVP）** 在 2026-02-13 完成并发布，同日或随后即推进 Phase 2/3；当前项目已到 v4.0.0，你所述「V1 阶段开发完成、正在做测试」应指 **MVP 功能已完成，处于 V1 的测试/收尾阶段**。

### 2.2 设计阶段（来自 design 文档）

- **角色**：R4（UI/UX 设计师）产出设计规范与 UI 方案；C2（开发）按规范实现。
- **产出**：
  - **design_system.md**：设计规范 v1.0（颜色、字体、间距、组件、布局、图标），与 Streamlit 能力对齐。
  - **ui_specs.md**：三个关键页面（问答页、知识库管理、文档上传）的布局、组件、交互逻辑，明确「请 C2 严格按照此规范实现」。
- **特点**：有设计文档和角色分工，但文档中**未体现**「需求澄清→多方案对比→用户确认」的成文流程，也未出现 `docs/plans/` 下的实施计划文档。

### 2.3 开发与测试阶段（来自 tests/README 与项目结构）

- **测试分期**：按 **Phase 1.1 → 1.2 → 1.3 → 1.4** 组织：
  - **Phase 1.1**：认证模块（Postgres、init_db、pytest）。
  - **Phase 1.2**：知识库与文档管理（可选 ChromaDB、测试文档生成脚本）。
  - **Phase 1.3**：RAG 问答（可选 LLM API，自动化通过 mock）。
  - **Phase 1.4**：MVP 安全与性能（全量回归、安全测试、性能测试，对应 test_security.py、test_performance.py）。
- **运行方式**：以 `pytest tests/ -v` 为主；Phase 1.4 明确「全量回归 + 安全 + 性能」。
- **E2E**：tests/e2e/human_simulator 提供基于 Playwright 的「模拟真人」流程（登录→知识库→上传→问答），有独立 README 说明安装与运行。
- **缺失**：tests/README 中引用的各 Phase 测试用例文档与测试报告（Phase_1.x_*.md）在仓库中未找到，无法还原当时的具体用例列表与报告结论。

### 2.4 工具引入阶段（V1 完成后）

- **安装方案**：评估 ECC、Superpowers 在 Cursor 中的可行性，选定方案 B（ECC + Superpowers），并给出 PowerShell 安装步骤与 superpowers-workflow 规则。
- **评估报告**：梳理 ECC 规则/代理/命令/技能与 Superpowers 技能，并做与 enterprise_rag 的优化映射。
- **保姆式使用说明**：面向零基础用户，把「优化项目」拆成五步，每步对应一句可复制发送给 AI 的提示（安全审查→代码审查→测试与覆盖率→文档更新→发布前验证）。
- **全盘模拟对照**：把项目活动与 ECC/Superpowers 能力做对照，并给出当前阶段（V1 测试期）的使用顺序建议。

上述四份文档均形成于 **V1 开发完成之后**，用于「如何用 ECC/Superpowers 优化/对照现有项目」，而非「从零用 ECC/Superpowers 做 V1」的过程记录。

综合上述文档，可归纳出你实际采用的开发逻辑与过程特征：

- 有 **R4/C2 角色分工** 和 **设计规范 + UI 方案**（design_system、ui_specs），说明存在「先设计再开发」的意识。

- 后端按 **api / core / models / schemas / services** 等分层，与 README 描述一致；前端为 Streamlit，符合设计文档。

- **安全、代码规范、覆盖率、文档更新、发布前验证** 的成文流程，是在 **引入 ECC/Superpowers 之后** 才以「保姆式五步」和「全盘模拟对照」的形式出现；V1 开发当时是否有系统化的安全审查、代码审查、覆盖率目标、发布前检查清单，无法从现有 .md 中还原。
